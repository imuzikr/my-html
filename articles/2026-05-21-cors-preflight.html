Preflight
브라우저가 먼저 허락을 구하는 이유

fetch() 한 번 날렸는데 네트워크 탭에 요청이 두 개? 내가 보낸 적 없는 OPTIONS가 먼저 나간다? 당신 잘못이 아닙니다. 브라우저가 자동으로 하는 보안 절차입니다.

1이런 경험, 있으신가요?

React나 Next.js로 프론트엔드를 만들고, 백엔드 API에 fetch()로 데이터를 요청했더니 콘솔에 이런 에러가 뜹니다.

Browser Console 
Access to fetch at 'https://api.myserver.com/data'
from origin 'http://localhost:3000' has been blocked
by CORS policy: Response to preflight request doesn't
pass access control check.

그리고 네트워크 탭을 열어보면 이상한 일이 벌어져 있습니다.

요청이 두 개입니다. 내가 POST를 한 번 보냈는데, OPTIONS가 먼저 나갔습니다. 이 OPTIONS가 바로 Preflight 요청입니다.

핵심

Preflight는 내가 만든 코드가 아닙니다. 브라우저가 보안상 필요하다고 판단할 때 자동으로 보내는 사전 확인 요청입니다. JavaScript 코드에서 OPTIONS를 직접 작성하지 않아도 됩니다.

2Preflight란 무엇인가

브라우저는 출처(Origin)가 다른 서버에 "위험할 수 있는 요청"을 보내기 전에, 먼저 그 서버에게 허락을 구합니다. 이 허락 요청이 Preflight(사전 비행)입니다.

출처(Origin)가 다르다는 것은?

프론트엔드: http://localhost:3000
백엔드 API: https://api.myserver.com

도메인, 프로토콜, 포트 중 하나라도 다르면 "다른 출처(Cross-Origin)"입니다. 로컬 개발 중에도 포트 번호가 달라지면 다른 출처로 간주합니다. (:3000 vs :8000)

브라우저가 이렇게 동작하는 이유는 보안 때문입니다. 만약 어떤 악성 사이트가 사용자의 브라우저를 통해 은행 API에 마음대로 요청을 보낼 수 있다면? Preflight는 서버가 "허용된 출처"를 명확히 선언하도록 강제해서 이를 방지합니다.

쉽게 비유하면 이렇습니다:

서버라는 건물에 들어가려면 먼저 입구의 문지기(서버)에게 "나 들어가도 되나요?"를 물어봐야 합니다. 문지기가 OK하면 그때 실제로 안으로 들어갑니다.

3언제 발생하나요?

모든 요청에 Preflight가 발생하지는 않습니다. 브라우저는 "단순하고 안전하다"고 판단되는 요청은 바로 보냅니다. 복잡하거나 위험할 수 있는 요청 앞에서만 Preflight를 먼저 보냅니다.

구분 단순 요청 — Preflight 없음 비단순 요청 — Preflight 발생
HTTP 메서드 GET, POST, HEAD PUT, DELETE, PATCH
Content-Type text/plain
application/x-www-form-urlencoded
multipart/form-data		 application/json ← 바이브 코더 99%
커스텀 헤더 없음 Authorization, X-Custom-Header 등
바이브 코더가 가장 자주 마주치는 조합

아래 코드 한 줄이 Preflight를 발생시키는 가장 흔한 패턴입니다. Content-Type: application/json이 "단순하지 않은" 요청으로 분류되기 때문입니다.

JavaScript 
// 이 코드는 Preflight를 발생시킵니다
await fetch('https://api.myserver.com/data', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',  // ← 이것 때문에!
    'Authorization': `Bearer ${token}`         // ← 이것도!
  },
  body: JSON.stringify({ name: '홍길동' })
});
POST인데 왜 Preflight가?

GET이나 POST라도 Content-Type: application/json이나 Authorization 헤더가 있으면 "비단순 요청"으로 분류됩니다. 메서드만 보는 게 아닙니다.

4어떻게 동작하나요?

Preflight는 총 3단계로 동작합니다. 브라우저와 서버 사이의 "대화"입니다.

1
브라우저 → 서버: "이런 요청 보내도 되나요?"
브라우저가 자동으로 OPTIONS 메서드로 Preflight를 보냅니다. "이 출처(Origin)에서 POST 메서드를 쓰고, Authorization · Content-Type 헤더를 사용할 건데 허용하나요?"라고 묻는 것입니다.
2
서버 → 브라우저: "허용합니다" (또는 거부)
서버가 Access-Control-Allow-* 헤더들로 응답합니다. 허용된 출처, 메서드, 헤더를 명시합니다. 이 응답이 없거나 허용 목록에 없으면 CORS 에러가 발생합니다.
3
브라우저 → 서버: 이제 실제 요청 전송
Preflight가 성공해야만 내가 만든 실제 요청이 전송됩니다. Preflight가 실패하면 브라우저는 실제 요청을 차단하고 콘솔에 CORS 에러를 표시합니다.

5서버-서버 통신에서는 Preflight가 없다

중요한 포인트입니다. Preflight와 CORS는 브라우저 환경에서만 발생합니다. 서버에서 서버로 직접 HTTP 요청을 보내는 경우에는 해당 없습니다.

브라우저 환경 — Preflight 발생

브라우저 보안 정책(CORS)의 영향을 받습니다.

  • React 앱 → 백엔드 API
  • Next.js 클라이언트 → 외부 API
  • Vue.js 앱 → REST API
  • 브라우저에서 직접 fetch()
서버 환경 — Preflight 없음

브라우저가 없으므로 CORS 정책이 적용되지 않습니다.

  • Next.js 서버 → 외부 API
  • Express 서버 → OpenAI API
  • Python 백엔드 → Firebase
  • GitHub Actions → 외부 서비스
실전 팁

프론트엔드에서 직접 외부 API를 호출할 때 CORS 에러가 난다면, 해당 호출을 Next.js API Routes나 백엔드 서버로 옮기는 것만으로 해결됩니다. 서버-서버 통신은 Preflight가 없으니까요.

6자주 빠지는 함정 4가지

①
서버가 OPTIONS 메서드를 모른다
Express.js에서 app.get(), app.post()만 만들고 OPTIONS를 처리하지 않으면 Preflight가 404나 405로 응답합니다. 에러 메시지는 CORS 에러로 표시되지만 진짜 원인은 OPTIONS 라우트 미등록입니다.
②
허용 헤더 목록에 실제 헤더가 빠져있다
Access-Control-Allow-Headers에 authorization을 빠뜨리면, Authorization 헤더를 쓰는 요청은 Preflight를 통과하지 못합니다. 실제로 보내는 헤더와 허용 목록이 정확히 일치해야 합니다.
③
Nginx나 API Gateway가 OPTIONS를 가로챈다
서버 코드는 올바른데 앞단의 Nginx, AWS API Gateway, Cloudflare가 OPTIONS 요청을 백엔드로 전달하지 않고 막아버리는 경우입니다. 서버 코드 외에 인프라 설정도 확인해야 합니다.
④
매 요청마다 Preflight가 나가서 느리다
Access-Control-Max-Age 헤더를 설정하지 않으면 브라우저는 Preflight 결과를 캐싱하지 않습니다. 모든 요청 전마다 OPTIONS를 보내게 됩니다. Access-Control-Max-Age: 86400을 추가하면 24시간 동안 캐싱됩니다.

7해결하기

서버 CORS 설정 — Express.js (Node.js)

가장 간단한 방법은 cors 패키지를 사용하는 것입니다.

JavaScript — Express.js 
const cors = require('cors');

app.use(cors({
  origin: ['http://localhost:3000', 'https://myapp.com'],
  methods: ['GET', 'POST', 'PUT', 'DELETE', 'PATCH', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  maxAge: 86400  // Preflight 결과 24시간 캐싱
}));

서버 CORS 설정 — Python FastAPI

Python — FastAPI 
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000", "https://myapp.com"],
    allow_methods=["*"],
    allow_headers=["*"],
    max_age=86400,
)

점검 체크리스트

CORS 에러가 났을 때 순서대로 확인하세요. 클릭하면 체크됩니다.

  • ✓
    서버에서 OPTIONS 메서드를 처리하고 있나요?
    cors 미들웨어를 추가했거나, OPTIONS 라우트가 등록되어 있어야 합니다.
  • ✓
    Access-Control-Allow-Origin에 요청 출처가 포함되나요?
    localhost:3000에서 요청한다면 허용 목록에 http://localhost:3000이 있어야 합니다.
  • ✓
    Access-Control-Allow-Headers에 실제로 보내는 헤더가 모두 있나요?
    Authorization, Content-Type 등 fetch()에서 사용하는 헤더를 빠짐없이 포함해야 합니다.
  • ✓
    Access-Control-Allow-Methods에 실제 HTTP 메서드가 있나요?
    PUT, DELETE, PATCH를 사용한다면 허용 목록에 포함해야 합니다.
  • ✓
    Nginx/API Gateway 같은 앞단 서버도 OPTIONS를 통과시키나요?
    서버 코드가 맞아도 인프라 레이어에서 막힐 수 있습니다.
  • ✓
    (선택) Access-Control-Max-Age로 캐싱을 설정했나요?
    86400 (24시간) 정도면 충분합니다. 매 요청마다 OPTIONS가 나가는 것을 방지합니다.
AI에게 물어볼 때 이렇게 말하세요

"Express.js 서버에서 CORS 설정을 추가해줘. 허용 출처는 http://localhost:3000이고, POST/PUT/DELETE 메서드와 Authorization, Content-Type 헤더를 허용해야 해. Preflight 캐싱도 24시간으로 설정해줘."

✓핵심 요약

  • 🌐
    Preflight = 브라우저가 먼저 보내는 허락 요청
    내 코드가 보내는 게 아닙니다. 브라우저가 CORS 보안 정책에 따라 자동으로 OPTIONS 요청을 먼저 보내서 서버의 허용 여부를 확인합니다.
  • ⚡
    application/json + Authorization = Preflight 발생
    바이브 코더가 가장 자주 쓰는 조합입니다. Content-Type: application/json이나 Authorization 헤더가 있으면 반드시 Preflight가 발생합니다.
  • 🖥️
    서버-서버 통신에는 Preflight 없음
    CORS는 브라우저 보안 정책입니다. Next.js API Routes나 백엔드 서버에서 외부 API를 호출하면 Preflight 없이 자유롭게 통신합니다.
  • 🔧
    해결은 항상 서버 쪽에서
    CORS 에러는 프론트엔드 코드를 수정해서 고칠 수 없습니다. 서버에서 Access-Control-Allow-* 헤더를 올바르게 설정하거나, cors 미들웨어를 추가해야 합니다.
