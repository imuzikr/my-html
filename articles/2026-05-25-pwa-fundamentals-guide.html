Section 01

PWA란 무엇인가?

PWA(Progressive Web App)는 웹 기술로 만들지만, 기기에 설치하고 앱처럼 실행할 수 있는 웹 애플리케이션입니다. 별도 앱스토어 없이 브라우저를 통해 배포됩니다.

💡
핵심 정의

구글 web.dev는 PWA를 "웹 기술로 만들지만 더 안정적이고 설치 가능한 앱 경험을 제공하는 웹 앱"으로 설명합니다. HTML/CSS/JS만으로 모바일 앱 수준의 사용자 경험이 가능합니다.

⚡

🔔
푸시 알림
🖥️
전체 화면 실행
Section 02

PWA의 3가지 핵심 요소

PWA는 기존 웹 앱에 두 가지 파일을 추가하는 것만으로 완성됩니다. 각 요소가 어떤 역할을 담당하는지 살펴봅니다.

① 기반
🌐
일반 웹 앱

HTML, CSS, JavaScript로 만든 웹사이트. 이 부분은 기존 웹사이트와 완전히 동일합니다.

index.html · style.css · app.js
② 신원 정보
🪪
Web App Manifest

앱 이름, 아이콘, 시작 주소, 배경색 등을 정의하는 JSON 파일. 브라우저에게 "이 앱을 설치할 수 있게 해 달라"고 알려줍니다.

manifest.json
③ 핵심 엔진
⚙️
Service Worker

브라우저 뒤에서 독립적으로 실행되는 JS 파일. 네트워크 요청을 가로채고 캐시를 관리해 오프라인 동작을 가능하게 합니다.

service-worker.js

manifest.json — 앱의 신원 카드

브라우저가 이 파일을 읽고 앱 이름, 아이콘, 실행 방식을 결정합니다. 반드시 HTML의 <head> 안에서 연결해야 합니다.

JSON manifest.json 
{
  "name": "나의 첫 PWA",
  "short_name": "MyPWA",
  "start_url": "/",
  "display": "standalone",  /* 브라우저 UI 없이 전체 화면 실행 */
  "background_color": "#ffffff",
  "theme_color": "#2563eb",
  "icons": [
    {
      "src": "/icon-192.png",
      "sizes": "192x192",
      "type": "image/png"
    },
    {
      "src": "/icon-512.png",
      "sizes": "512x512",
      "type": "image/png"
    }
  ]
}
HTML index.html — manifest 연결 
<head>
  <link rel="manifest" href="/manifest.json" />
</head>
Section 03

Service Worker — PWA의 심장

Service Worker는 브라우저와 네트워크 사이에서 모든 HTTP 요청을 중간에서 처리하는 독립적인 JavaScript 프로세스입니다.

Service Worker 동작 원리 — 캐시 vs. 네트워크 판단 흐름
👤
사용자 (브라우저)
index.html 불러와 줘!
⚙️
Service Worker (중간 관리자)
잠깐, 내가 먼저 캐시 창고를 확인해 볼게.
🔍 fetch 이벤트 가로채기
📦
Cache Storage
있어! index.html 여기 있거든.
⚙️
Service Worker
캐시에 있으니까 인터넷 안 써도 돼.
바로 응답할게! ✓ 오프라인 가능
⚙️
Service Worker (캐시 없을 때)
캐시에 없으면 인터넷에서 직접 가져오고 다음을 위해 저장해 둬.
⚠️
중요 조건

Service Worker는 HTTPS 환경에서만 동작합니다. 로컬 개발 시에는 localhost는 예외적으로 허용됩니다.

Section 04

처음 접속부터 오프라인까지

사용자가 PWA에 처음 접속할 때 어떤 일이 순서대로 일어나는지 확인합니다.

1
웹사이트 최초 접속
사용자가 URL을 입력하거나 링크를 클릭합니다.
2
index.html 로드
서버에서 HTML 파일을 받아옵니다. manifest.json도 링크로 연결됩니다.
3
app.js 실행 → Service Worker 등록
navigator.serviceWorker.register()가 호출되어 SW를 브라우저에 등록합니다.
4
install 이벤트 — 필수 파일 캐싱
SW가 처음 설치될 때 지정한 파일 목록(FILES_TO_CACHE)을 Cache Storage에 저장합니다.
5
activate 이벤트 — SW 활성화
구버전 캐시를 정리하고 SW가 네트워크 요청을 처리할 준비를 마칩니다.
6
다음 접속부터 — 캐시 활용
모든 요청이 SW를 거치며 캐시에서 즉시 응답합니다. 로딩 속도가 크게 빨라집니다.
7
인터넷 끊겨도 동작
네트워크가 없어도 캐시에 저장된 파일로 화면이 표시됩니다. 오프라인 경험이 완성됩니다.
Section 05

최소 코드 예시

PWA를 구성하는 최소한의 파일 구조와 각 파일의 핵심 코드입니다.

my-pwa/
index.html ← 메인 HTML
app.js ← Service Worker 등록
service-worker.js ← 캐시 관리 핵심
manifest.json ← 앱 정보
icon-192.png ← 앱 아이콘 (소형)
icon-512.png ← 앱 아이콘 (대형)
HTML index.html 
<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <title>나의 PWA</title>
  <link rel="manifest" href="/manifest.json" />
</head>
<body>
  <h1>나의 첫 PWA</h1>
  <script src="/app.js"></script>
</body>
</html>
JavaScript app.js — Service Worker 등록 
if ("serviceWorker" in navigator) {
  navigator.serviceWorker
    .register("/service-worker.js")
    .then(() => {
      console.log("Service Worker 등록 성공");
    })
    .catch((error) => {
      console.error("Service Worker 등록 실패:", error);
    });
}
JavaScript service-worker.js — 핵심 캐싱 로직 
const CACHE_NAME = "my-pwa-cache-v1";
const FILES_TO_CACHE = [
  "/",
  "/index.html",
  "/app.js",
  "/manifest.json"
];

/* 설치 단계: 필수 파일을 캐시에 저장 */
self.addEventListener("install", (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => {
      return cache.addAll(FILES_TO_CACHE);
    })
  );
});

/* fetch 단계: 캐시 우선, 없으면 네트워크 */
self.addEventListener("fetch", (event) => {
  event.respondWith(
    caches.match(event.request).then((cachedFile) => {
      return cachedFile || fetch(event.request);
      /* ↑ 핵심: 캐시 있으면 캐시, 없으면 네트워크 */
    })
  );
});
✅
핵심 한 줄

return cachedFile || fetch(event.request) — 이 한 줄이 오프라인 동작의 전부입니다. 캐시에 있으면 캐시를, 없으면 네트워크를 사용합니다.

Section 06

일반 웹사이트 vs. PWA

기존 웹사이트와 PWA가 실제로 어떻게 다른지 항목별로 비교합니다.

항목 일반 웹사이트 PWA
홈 화면 설치 불가능 가능 (manifest.json)
오프라인 동작 거의 불가능 가능 (Service Worker + Cache)
로딩 속도 매번 서버에서 다운로드 캐시에서 즉시 응답
실행 방식 브라우저 UI 포함 standalone (앱처럼 전체 화면)
푸시 알림 불가 가능 (Push API)
캐시 제어 브라우저 기본 캐시 (개발자 제어 불가) 개발자가 직접 전략 설계
핵심 추가 파일 없음 manifest.json + service-worker.js
앱스토어 필요 여부 해당 없음 불필요 (브라우저로 배포)
Section 07

쉬운 비유 — 학교 매점으로 이해하기

PWA의 각 구성 요소를 학교 매점에 비유하면 더 직관적으로 이해할 수 있습니다.

🏪
웹 앱 (HTML/CSS/JS)
학교 매점 그 자체

물건을 파는 공간. 인터넷이 있으면 언제든 서버에서 콘텐츠를 받아올 수 있습니다.

🪧
manifest.json
매점 간판과 앱 아이콘

매점 이름, 위치, 운영시간 등 "이게 뭔지"를 알려주는 정보. 홈 화면의 앱 아이콘과 이름이 됩니다.

👨‍💼
Service Worker
물건 미리 챙기는 매점 관리자

재고를 미리 파악하고 창고에 인기 상품을 쌓아 두는 사람. 모든 요청을 먼저 받아 처리합니다.

📦
Cache Storage
미리 쌓아 둔 창고

납품이 끊겨도(인터넷 단절) 창고에서 꺼내 팔 수 있습니다. 오프라인 동작의 비결입니다.

🎯
비유 결론

인터넷이 끊겨도 Service Worker(관리자)가 Cache(창고)에서 필요한 파일을 꺼내 주기 때문에, 최소한의 화면은 계속 보여줄 수 있습니다.

Section 08

핵심 정리

PWA의 모든 것을 한 문장으로 요약하면 아래 공식입니다.

PWA = 웹 앱 + Manifest + Service Worker

🌐 웹 앱
HTML · CSS · JS
+
🪪 manifest.json
앱 신원 · 설치
+
⚙️ service-worker.js
캐시 · 오프라인
=
📲 PWA
앱처럼 동작하는 웹

Service Worker가 캐시를 관리하기 때문에 오프라인 실행과 빠른 로딩이 가능해지며, Manifest 덕분에 홈 화면에 설치할 수 있습니다.

🌍
일반 웹사이트
"브라우저로만 접근하는 전통적인 웹"
  • 항상 인터넷 필요
  • 브라우저 UI 포함
  • 홈 화면 설치 불가
  • 캐시 개발자 제어 불가
📲
PWA
"앱처럼 설치되고 작동하는 웹"
  • 오프라인에서도 동작
  • 전체 화면 앱처럼 실행
  • 홈 화면에 설치 가능
