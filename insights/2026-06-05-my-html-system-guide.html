Repository Guide

my-html 저장소 구조와 디자인 시스템 해설

이 문서는 `imuzikr/my-html` 저장소를 기준으로, HTML 콘텐츠가 어떻게 작성되고, 어떤 디자인 토큰을 사용하며, GitHub Actions가 어떤 자동화 흐름을 담당하는지 운영 관점에서 정리한 가이드입니다.

읽은 핵심 파일
  • `scripts/generate_index.py` - 인덱스 자동 생성
  • `design-system.html` - 시각 언어와 컴포넌트 기준
  • `html-skill/SKILL.md` - HTML 생성 규칙
  • `.github/workflows/deploy.yml` - 배포 자동화
Core Summary

이 저장소는 단순히 HTML 파일을 모아두는 곳이 아니라, 콘텐츠 HTML 작성 규칙, 디자인 시스템, 인덱스 생성 스크립트, GitHub Pages 배포 자동화가 결합된 작은 HTML 퍼블리싱 파이프라인입니다. 사람이 직접 추가해야 하는 것은 주로 `insights/` 또는 `vibe/`의 새 HTML 콘텐츠이고, 목록 페이지와 서비스 워커 캐시 버전은 자동화가 갱신합니다.

1

저장소는 세 층으로 나뉩니다

콘텐츠, 규칙, 자동화가 분리되어 있어야 오래 관리할 수 있습니다.

Content

`insights/`, `vibe/`, `assets/`

실제 독자가 보는 HTML 콘텐츠와 이미지 자산이 들어갑니다. 새 글을 추가할 때는 자동 생성 파일보다 이 콘텐츠 폴더를 건드리는 것이 핵심입니다.

Rules

`design-system.html`, `html-skill/`

어떤 색, 간격, 카드, 탭, 코드 블록, 반응형 규칙을 써야 하는지 정의합니다. 새 HTML은 이 규칙을 따라야 목록과 톤이 맞습니다.

Automation

`scripts/generate_index.py`

HTML 파일을 스캔하고 메타데이터를 읽어 루트 인덱스와 카테고리별 인덱스를 다시 만듭니다. 뒤로가기 버튼과 모바일 폰트 보정도 주입합니다.

Delivery

`.github/workflows/deploy.yml`

`main` 브랜치에 푸시되면 인덱스 생성, 생성 파일 커밋, GitHub Pages 배포까지 이어지는 자동화 흐름을 담당합니다.

2

`generate_index.py`의 역할

이 스크립트는 목록 페이지를 손으로 관리하지 않게 해주는 중심 장치입니다.

스크립트는 알려진 카테고리 폴더를 기준으로 HTML 파일을 수집합니다. 현재 구조에서는 `insights`와 `vibe`가 주요 카테고리입니다. 각 HTML에서 `<title>`, `<meta name="description">`, 파일명 앞 날짜, `og:image`를 읽어 카드형 목록에 필요한 데이터를 만듭니다.

Python · 핵심 수집 방식 
def collect_files():
    # 알려진 카테고리 폴더만 수집합니다.
    for category in KNOWN_CATEGORIES:
        folder_path = ROOT / category
        for html in sorted(folder_path.glob("*.html"), reverse=True):
            files.append({
                "title": get_title(html),
                "description": get_description(html),
                "date": get_date(html),
                "og_image": get_og_image(html),
            })
스크립트가 직접 해주는 일
  • 카테고리별 HTML 파일을 모아 카드 목록 데이터를 만듭니다.
  • `insights/index.html`, `vibe/index.html`, 루트 `index.html`을 다시 생성합니다.
  • 각 콘텐츠 HTML에 목록으로 돌아가는 버튼이 없으면 자동으로 삽입합니다.
  • 모바일에서 글자가 너무 작아지지 않도록 폰트 스케일 CSS를 주입합니다.
  • `sw.js`의 캐시 버전을 갱신해 새 배포가 브라우저에 반영되도록 돕습니다.
3

디자인 시스템은 토큰 중심입니다

색, 간격, 글자 크기, 그림자, 반경을 변수로 고정해 HTML마다 같은 인상을 유지합니다.

`design-system.html`과 `html-skill/SKILL.md`는 새 HTML이 반드시 동일한 `:root` 토큰 블록에서 출발하도록 요구합니다. 이 규칙의 장점은 콘텐츠를 많이 만들더라도 배경색, 카드 반경, 강조색, 텍스트 위계가 흔들리지 않는다는 점입니다.

색상은 배경, 본문, 보조 텍스트, 테두리, 강조색, 의미 색상으로 나뉩니다. `--color-accent`는 주요 액션과 하이라이트에 쓰이고, `--color-blue`, `--color-green`, `--color-yellow`, `--color-red`는 상태와 의미 구분에 사용됩니다.

간격은 `--space-1`부터 `--space-12`까지의 스케일을 사용합니다. 새 컴포넌트에서 임의의 픽셀값을 늘리는 대신 이 스케일을 쓰면 페이지 전체 리듬이 유지됩니다.

문서에서 반복되는 기본 부품은 카드, 탭, 테이블, 코드 블록, 배지, 버튼, 콜아웃, 세부 접기 영역입니다. 새 HTML을 만들 때도 이 부품을 조합하면 저장소의 기존 글과 어긋나지 않습니다.

긴 문서는 sticky nav나 목차를 두고, 3개 이상의 주요 섹션은 탭을 쓰라는 규칙이 있습니다. 이 문서도 상단 목차와 탭 패널을 함께 사용합니다.

호버는 `transition: all 180ms ease`와 작은 lift를 사용하고, 탭 같은 상태 전환은 `140ms ease` 수준으로 짧게 유지합니다. 모든 버튼과 입력 대상은 모바일에서 누르기 쉽게 최소 44px 높이를 확보해야 합니다.

코드 복사 기능은 `navigator.clipboard`를 먼저 사용하고, 실패하면 `execCommand` 방식으로 대체하는 패턴을 사용합니다.

영역 대표 토큰/패턴 운영 의미
Foundation `--color-bg`, `--color-text`, `--font-sans` 모든 문서가 같은 배경, 본문 색, 글꼴 체계를 공유합니다.
Components `.card`, `.tabs`, `.code-block`, `.callout` 새 문서도 같은 부품을 써서 일관된 읽기 경험을 제공합니다.
Responsive 768px 단일 컬럼, 880px 이하 사이드바 숨김 데스크톱 문서 구조가 모바일에서 깨지지 않도록 합니다.
Restriction 외부 CDN, 외부 이미지 URL 금지 HTML 파일 하나만으로 오프라인에서도 열리는 결과물을 지향합니다.
4

배포 흐름은 GitHub Actions가 담당합니다

사람은 콘텐츠를 추가하고, 자동화는 목록과 배포를 갱신합니다.

`.github/workflows/deploy.yml`은 `main` 브랜치에 푸시될 때 실행됩니다. 체크아웃 후 `python3 scripts/generate_index.py`를 실행하고, 변경된 자동 생성 파일이 있으면 `github-actions[bot]` 이름으로 다시 커밋합니다. 이후 GitHub Pages 아티팩트를 업로드하고 배포합니다.

i
핵심 운영 원칙 콘텐츠 HTML은 사람이 만들지만, 목록 페이지와 캐시 버전은 자동화가 관리합니다. 그래서 `index.html`, `insights/index.html`, `vibe/index.html`, `sw.js`를 직접 고치는 일은 피하는 편이 좋습니다.
GitHub Actions가 하는 일을 순서대로 보기
  • `actions/checkout`으로 저장소를 가져옵니다.
  • `scripts/generate_index.py`를 실행해 목록 페이지를 재생성합니다.
  • 생성된 변경분이 있으면 `[skip ci]` 메시지로 커밋해 반복 실행을 막습니다.
  • GitHub Pages 설정을 준비하고 전체 저장소를 아티팩트로 업로드합니다.
  • `actions/deploy-pages`가 실제 Pages 배포를 완료합니다.
5

새 HTML을 추가할 때의 실전 체크리스트

이 저장소에서는 파일을 어디에 두는지가 자동화 결과를 결정합니다.

  • 카테고리를 먼저 정합니다. 분석 글은 `insights/`, 감각적 개발 글은 `vibe/`에 두는 흐름입니다.
  • 파일명은 날짜로 시작합니다. 예: `2026-06-05-my-html-system-guide.html`. 스크립트가 파일명에서 날짜를 읽습니다.
  • `title`과 `description`을 꼭 넣습니다. 인덱스 카드의 제목과 설명으로 사용됩니다.
  • 디자인 토큰을 사용합니다. CSS에서 색과 간격을 임의값으로 흩뿌리지 말고 `var(--...)`를 우선 사용합니다.
  • 자동 생성 파일은 건드리지 않습니다. 인덱스와 서비스 워커 캐시 갱신은 Actions에 맡기는 것이 안전합니다.
Shell · 권장 작업 흐름 
# 1. 새 HTML 파일만 콘텐츠 폴더에 추가
git add insights/2026-06-05-my-html-system-guide.html

# 2. 콘텐츠 변경만 커밋
git commit -m "Add my-html system guide"

# 3. main에 푸시하면 GitHub Actions가 인덱스와 배포를 처리
git push origin main

결론

`my-html` 저장소의 본질은 HTML 파일 모음이 아니라, 사람이 쓴 단일 HTML 문서를 디자인 시스템과 자동화 스크립트로 일관되게 공개하는 파이프라인입니다. 좋은 운영 방식은 새 콘텐츠만 명확히 추가하고, 목록 생성과 배포는 자동화에 맡기는 것입니다.

작성 기준: `scripts/generate_index.py`, `design-system.html`, `html-skill/SKILL.md`, `.github/workflows/deploy.yml` 구조 확인.
