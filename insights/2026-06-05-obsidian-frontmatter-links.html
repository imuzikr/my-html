Obsidian Note System

옵시디언은 분류보다 연결이 먼저입니다

uxdev.org의 글은 옵시디언을 처음 쓸 때 흔히 빠지는 함정을 짚습니다. 폴더와 태그를 촘촘하게 설계하기보다, 실제로 함께 읽어야 하는 노트들을 `[[링크]]`로 연결하고 프론트매터는 필요한 구조화 정보만 담으라는 관점입니다.

핵심 결론
  • 노트 간 실제 관계는 위키 링크로 연결합니다.
  • 태그는 넓은 분류에만 사용합니다.
  • 상태, 생성일, 유형은 프론트매터에 둡니다.
  • 반복 입력은 템플릿으로 자동화합니다.
Source-Based Summary

원문은 옵시디언 사용자가 처음 고민하는 두 가지 문제, 즉 노트 연결성과 프론트매터 정리법을 함께 다룹니다. 핵심은 프론트매터나 태그 체계를 먼저 과도하게 설계하지 말고, 본문 안의 `[[위키 링크]]`를 통해 지식이 자연스럽게 이어지도록 만드는 것입니다.

1

프론트매터보다 링크 구조가 먼저입니다

원문은 옵시디언의 강점을 개별 문서 저장이 아니라 노트 네트워크에 둡니다.

글은 초보 사용자가 속성, 태그, 분류 체계를 먼저 세밀하게 만들려는 경향을 경계합니다. 옵시디언에서 연결성을 높이는 직접적인 도구는 `[[위키 링크]]`입니다. 예를 들어 프로젝트 관리 글에서 `[[옵시디언 템플릿]]`, `[[개인 지식 관리]]` 같은 관련 노트를 본문에 연결하면, 읽는 사람은 문맥을 따라 자연스럽게 이동할 수 있습니다.

i
원문에 근거한 판단 `CSS`, `TailwindCSS`, `디자인 시스템`, `컴포넌트 재사용`처럼 실제 맥락에서 서로 이어지는 개념은 태그만 붙이기보다 본문 링크로 연결하는 편이 더 유용하다고 설명합니다.
2

프론트매터는 최소한으로 시작합니다

많은 필드를 처음부터 만들면 쓰는 사람이 지칩니다.

프론트매터는 노트 맨 위에 들어가는 YAML 메타데이터입니다. 제목, 태그, 생성일, 상태처럼 일정한 형식으로 관리하고 싶은 정보를 담습니다. 다만 글은 처음부터 많은 필드를 넣는 방식이 오히려 지속성을 떨어뜨린다고 봅니다.

YAML · 최소 시작 예시 
---
type: note
status: active
tags: []
created: 2026-04-15
related: []
---
항목 역할 사용 기준
type 노트 유형 일상 메모, 프로젝트, 참고 자료, 허브 노트를 구분합니다.
status 진행 상태 active, draft, archived처럼 운영 상태를 관리할 때 씁니다.
tags 넓은 범주 관계 표현보다 project, reference, idea 같은 큰 분류에 적합합니다.
related 명시적 관련 노트 본문 링크와 별도로 관계를 더 분명히 관리하고 싶을 때 씁니다.
3

노트 역할에 따라 다르게 운영합니다

모든 노트에 같은 속성을 강제하지 않는 것이 원문의 실전 조언입니다.

일상 메모나 짧은 아이디어는 프론트매터를 최소화합니다. 핵심은 빨리 기록하는 것이므로 `type`과 `created` 정도면 충분합니다.

프로젝트 노트는 `status`, `project`, `related`가 유용합니다. 진행 중인 작업과 관련 노트를 함께 묶어 추적할 수 있기 때문입니다.

참고 자료 노트는 `topic`, `source`, `created`처럼 출처와 주제를 확인할 수 있는 정보가 중요합니다.

MOC는 Map of Contents입니다. 프론트매터는 최소화하고, 본문 안에 관련 노트 링크를 목록으로 정리해 주제별 길잡이 역할을 하게 합니다.

  • 실제 관계는 링크로 연결합니다. 함께 읽어야 하는 노트는 본문에서 직접 이어줍니다.
  • 태그는 넓게 씁니다. 태그를 관계 표현 도구로 쓰면 금방 복잡해집니다.
  • 반복 입력은 템플릿으로 줄입니다. `created: {{date}}` 같은 값은 템플릿에 맡기는 편이 안정적입니다.
  • 노트가 많아지면 MOC를 만듭니다. 특정 주제의 중심 허브가 생기면 탐색 비용이 줄어듭니다.
4

플러그인은 기본 구조 이후의 보조 도구입니다

원문은 복잡한 플러그인보다 기본 링크 구조를 먼저 잡으라고 봅니다.

Templater

동적 템플릿 자동화

새 노트를 만들 때 날짜나 정해진 프론트매터를 자동으로 넣고 싶을 때 유용합니다.

Metadata Menu

속성 관리

프론트매터 필드가 많아졌을 때 여러 노트의 속성 값을 일관되게 관리하는 데 도움을 줍니다.

Dataview

조회 가능한 데이터

특정 상태의 프로젝트 노트만 모으거나 같은 태그 문서를 자동 목록화할 때 강력합니다.

MOC

주제별 허브

플러그인은 아니지만 관련 노트를 한곳에 묶어 탐색을 돕는 매우 실용적인 운영 방식입니다.

이 글을 실제 옵시디언 운영 규칙으로 바꾸면
  • 새 노트를 만들 때 먼저 관련 노트를 1개 이상 링크할 수 있는지 확인합니다.
  • 태그는 `#project`, `#reference`, `#idea`, `#draft`처럼 적은 수로 유지합니다.
  • 프론트매터 필드는 `type`, `status`, `created`, `related` 정도로 시작합니다.
  • 반복되는 프론트매터는 기본 템플릿이나 Templater로 자동화합니다.
  • 노트가 쌓이면 주제별 MOC 노트를 만들어 길잡이 역할을 맡깁니다.

결론

이 글의 메시지는 단순합니다. 옵시디언을 오래 쓰려면 정교한 분류표를 먼저 만들기보다, 실제 생각의 흐름을 따라 노트를 연결해야 합니다. 프론트매터는 그 연결을 대체하는 도구가 아니라, 상태와 날짜처럼 구조화가 필요한 정보를 보조하는 장치입니다.
