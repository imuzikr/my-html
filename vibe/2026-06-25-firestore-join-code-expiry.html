입장 코드를
2주 후에 만료시키기

교실 Q&A 앱에서 학생이 학급에 들어올 때 쓰는 입장 코드. 한 번 만들면 영원히 유효하다면, 유출된 코드로 1년 뒤에도 낯선 사람이 들어올 수 있습니다. 코드에 수명(2주)을 주되, 이미 들어온 학생은 건드리지 않는 설계를 다이어그램으로 풀어봅니다.

Firestore 보안 규칙 Cloud Functions defense-in-depth 2026-06-25
핵심 요약

입장 코드는 입장권, 멤버십은 정기권입니다. 코드를 문서 ID로 두면 아는 사람만 콕 집어 읽고(get) 긁어모으기(list)는 막힙니다. 여기에 expiresAt 필드로 2주짜리 수명을 주면, 유출된 코드도 시간이 지나면 스스로 무력화됩니다. 단, 만료는 신규 입장만 막아야 합니다. 이미 들어온 학생의 소속은 uid 기준 멤버십으로 따로 저장해, 코드가 만료돼도 기기를 바꿔도 그대로 유지됩니다. 그리고 활동 중 잠기는 사고를 막기 위해 관리자 원클릭 재발급을 함께 둡니다.

1

왜 입장 코드에 수명을 줘야 하나

입장 코드는 본질적으로 "아는 사람은 쓸 수 있는 비밀 값"입니다. 그런데 비밀은 시간이 지날수록 샙니다. 칠판에 적혔다 사진 찍히고, 단톡방에 공유되고, 졸업한 형이 후배에게 알려줍니다.

코드가 영구히 유효하면, 한 번이라도 유출된 코드는 영원히 위험합니다. 작년에 찍힌 칠판 사진 한 장으로 모르는 사람이 올해 우리 반에 들어올 수 있습니다. 해결의 핵심은 코드에 유효기간을 줘서, 설령 유출되더라도 시간이 지나면 스스로 무력화되게 만드는 것입니다.

🎟️
비유 — 입장권과 정기권

입장 코드는 놀이공원 당일 입장권입니다. 들어올 때 한 번 쓰고 나면 역할이 끝납니다. 반면 한 번 들어온 학생의 소속은 회원 정기권입니다. 입장권 유효기간이 지나도, 이미 회원이 된 사람은 계속 회원입니다. 이 둘을 섞으면 "입장권 만료 = 회원 추방"이라는 사고가 납니다. 분리하는 것이 설계의 핵심입니다.

이것이 보안에서 말하는 다층 방어(defense-in-depth)입니다. 코드를 문서 ID로 숨기는 1차 방어에 더해, 시간 제한이라는 2차 방어를 겹쳐 둡니다. 한 겹이 뚫려도 다음 겹이 막습니다.

영구 코드는 유출 즉시 영구 위험. 2주 만료 코드는 노출 창을 14일로 잘라낸다.

2

출발점: 코드를 "문서 ID"로 둔다

만료를 붙이기 전에, 코드 저장 방식부터 바로잡아야 합니다. 코드를 문서의 필드에 두면 만료를 붙여도 새는 구멍이 남습니다.

읽기에는 두 종류가 있습니다. get은 문서 ID를 정확히 알아야 그 문서 하나만 읽고(사물함 번호를 알아야 그 칸만 엶), list는 컬렉션을 쭉 훑어 가져옵니다(사물함 전체를 다 엶). 보안 규칙에서 이 둘을 따로 허용할 수 있다는 점이 설계의 지렛대입니다.

✗ 코드를 필드에 두면

classes/xJ3k9 { joinCode:"ABCDEF" }

ID는 의미 없는 자동 번호라, 코드로 찾으려면 전체 훑기(list)가 필요합니다. 훑는 김에 다른 반 코드까지 다 보입니다. "로그인하면 읽기 허용" 규칙이면 누구나 모든 코드를 긁어모을 수 있습니다.

✓ 코드를 문서 ID로 두면

joinCodes/ABCDEF { classId:"xJ3k9" }

코드를 아는 사람만 get("ABCDEF")로 그 문서 하나에 도달합니다. get은 누구나 허용, list는 교사만 허용으로 나누면, 모르는 사람은 다른 코드를 알아낼 길이 없습니다.

코드 = 문서 ID. get은 콕 집어 읽기(누구나), list는 전체 훑기(교사만). 침입자의 긁어모으기가 막힌다.

🔑
한 줄 정리

비밀 값을 필드에 두면 찾으려다 다 새고, 문서 ID로 두면 아는 사람만 콕 집어 읽고(get) 훑기(list)는 관리자만 → 우회 차단. 만료는 이 토대 위에 얹습니다.

3

만료 시각을 어떻게 저장하나

"2주 후 만료"를 데이터로 표현하는 방법은 두 가지입니다. 둘 중 만료 시각 자체를 저장하는 쪽이 검증이 단순합니다.

방식저장 형태만료 판정평가
A. 생성시각 + 기간 createdAt만 저장 읽을 때마다 createdAt + 14일 < now 계산 기간을 바꾸면 과거 코드까지 소급 적용됨 (장점이자 함정)
B. 만료시각 직접 저장 ✓ expiresAt 저장 expiresAt < now 한 번 비교 판정이 단순·명확. 코드마다 수명을 다르게 줄 수도 있음

권장은 B입니다. 코드를 만들 때 expiresAt = now + 14일을 함께 적어 둡니다. 그러면 "만료됐나?"는 expiresAt < now 단 한 번의 비교로 끝납니다. 보안 규칙 안에서도 쉽게 검사할 수 있습니다(뒤에서).

코드를 만들 때 expiresAt = createdAt + 14일을 함께 박아 둔다. 판정은 expiresAt < now 한 번.

javascript — 코드 생성 (서버/Cloud Function 권장)
// 14일을 밀리초로: 14 * 24 * 60 * 60 * 1000
const TWO_WEEKS = 14 * 24 * 60 * 60 * 1000;

const now = Timestamp.now();
const expiresAt = Timestamp.fromMillis(now.toMillis() + TWO_WEEKS);

await setDoc(doc(db, "joinCodes", code), {
  classId,
  createdAt: now,
  expiresAt,                // ← 만료 시각을 직접 저장
  createdBy: teacherUid,
});
4

입장 흐름: 코드 한 줄로 끝나지 않는다

학생이 코드를 입력하면 내부에서는 세 단계가 순서대로 일어납니다. 코드 존재 → 만료 → 멤버십 생성. 어느 한 단계라도 실패하면 입장이 막힙니다.

코드 입력 → get으로 존재 확인 → expiresAt 만료 확인 → 통과하면 멤버십 생성 → 입장.

javascript — 입장 검증의 뼈대
async function joinClass(code, uid) {
  const snap = await getDoc(doc(db, "joinCodes", code));

  // ② 존재 확인
  if (!snap.exists()) throw new Error("잘못된 코드입니다");

  // ③ 만료 확인 — expiresAt 과 현재 시각 비교
  const { classId, expiresAt } = snap.data();
  if (expiresAt.toMillis() < Date.now()) {
    throw new Error("만료된 코드입니다. 선생님께 새 코드를 받으세요");
  }

  // ④ 멤버십 생성 — uid 기준으로 소속을 기록
  await setDoc(doc(db, "memberships", `${uid}_${classId}`), {
    uid, classId, joinedAt: serverTimestamp(),
  });
}
⚠️
주의 — 이 코드는 "뼈대"일 뿐

위는 클라이언트에서 검증하는 단순 버전입니다. 작정한 사용자는 앱 화면을 우회해 memberships에 직접 쓸 수 있습니다. 진짜 안전하게 하려면 검증을 어디서 하느냐가 중요합니다 — 다음 섹션.

5

검증을 "어디서" 하나 — 3개의 방어선

같은 만료 검사라도 어디서 실행하느냐에 따라 안전 강도가 다릅니다. 클라이언트는 속이기 쉽고, 서버는 속일 수 없습니다.

클라이언트는 친절한 안내, 보안 규칙은 강제, Cloud Function은 우회 불가. 겹쳐 쌓는다.

🙂
간단 버전

"본인 것만 생성(uid 일치)" 규칙 + 코드·만료 검증은 앱에서. 일반 사용엔 충분, 작정하면 우회 가능.

🛡️
튼튼한 버전

입장을 Cloud Function으로 처리. 서버가 만료·코드 검증 후에만 멤버십 생성, 클라이언트 직접 쓰기 불가 → 우회 불가.

6

만료만 두면 생기는 사고 — 재발급을 함께

"무조건 자동 만료"만 두면, 정당한 사용자가 활동 중에 잠길 수 있습니다. 그래서 만료 + 관리자용 원클릭 재발급을 한 쌍으로 둡니다.

예를 들어 전학 온 학생이 2주가 막 지난 코드를 받았거나, 교사가 깜빡하고 코드를 갱신하지 않았다면, 새 학생이 들어올 길이 막힙니다. 보안(자동 만료)은 유지하면서 이 잠김을 즉시 푸는 장치가 재발급입니다.

발급 → 유효 → 만료의 순환에, 교사가 언제든 끼어들어 재발급할 수 있는 경로를 더한다.

🔁
재발급의 두 가지 방법

(가) expiresAt만 연장 — 같은 코드의 만료 시각을 now+14d로 갱신. 간단하지만 유출된 코드가 그대로 살아납니다. (나) 코드 교체 — 옛 문서를 지우고 새 코드 문서를 발급. 유출된 옛 코드를 무력화하므로 보안상 더 권장됩니다.

7

가장 중요한 분리 — 코드 만료 ≠ 학생 추방

여기서 실수하면 "코드가 만료됐더니 멀쩡히 활동하던 학생들이 다 튕겨 나갔다"는 사고가 납니다. 코드와 멤버십은 완전히 다른 데이터입니다.

입장 코드는 처음 들어올 때의 입장권이고, 멤버십은 한 번 들어오면 유지되는 소속 기록입니다. 코드 만료는 신규 입장만 막고, 기존 멤버는 건드리지 않습니다. 둘을 다른 컬렉션으로 저장하기 때문에 자연스럽게 분리됩니다.

코드는 입장 순간에만 쓰이고 14일 후 만료. 소속은 uid 기준 멤버십에 별도 저장되어 만료·기기변경과 무관하게 유지.

🪪
비유 — 신분증과 책상 위 포스트잇

uid는 어디서든 통하는 신분증(계정에 붙음, 기기 바뀌어도 동일). 서버의 멤버십은 중앙 사무실의 내 서류함(uid로 찾음). 반면 소속을 브라우저 localStorage에만 두면 그건 그 컴퓨터 책상의 포스트잇이라, 기기를 바꾸거나 캐시를 지우면 사라집니다. "기기를 넘어 유지"가 필요한 순간 서버에 uid로 옮겨야 합니다.

🚫
흔한 함정

소속 상태를 localStorage에만 저장하면 그 값은 그 기기·그 브라우저 안에서만 유지됩니다. 처음 구현할 때 간단해서 그렇게 두기 쉽지만, 학생이 학교 컴퓨터에서 집 노트북으로 바꾸는 순간 소속이 사라집니다. 유지돼야 할 상태는 서버에 uid 기준으로 저장하세요.

8

보안 규칙으로 못 박기

지금까지의 설계를 Firestore 보안 규칙으로 옮깁니다. 핵심은 세 가지 — 코드 get/list 분리, 만료 검사, 멤버십은 본인 uid만.

firestore.rules
rules_version = '2';
service cloud.firestore {
  match /databases/{db}/documents {

    // 입장 코드: get 은 로그인한 누구나, list 는 교사만
    match /joinCodes/{code} {
      allow get: if request.auth != null;
      allow list: if isTeacher();
      allow write: if isTeacher();   // 발급·재발급은 교사만
    }

    // 멤버십: 문서 ID 안의 uid 가 본인일 때만 생성
    match /memberships/{mid} {
      allow read: if request.auth != null
                  && resource.data.uid == request.auth.uid;
      allow create: if request.auth != null
        && request.resource.data.uid == request.auth.uid
        // ★ 만료 검사: 가리키는 코드가 아직 살아 있어야 함
        && get(/databases/{db}/documents/joinCodes/$(request.resource.data.code))
             .data.expiresAt > request.time;
    }

    function isTeacher() {
      return request.auth != null
        && get(/databases/{db}/documents/users/$(request.auth.uid))
             .data.role == "teacher";
    }
  }
}
⏱️
규칙 안의 시간 — request.time

보안 규칙에서 request.time은 서버가 요청을 받은 시각입니다. 클라이언트 시계를 조작해도 이 값은 못 바꿉니다. 그래서 expiresAt > request.time으로 만료를 검사하면, 사용자가 기기 시간을 과거로 돌려도 만료된 코드로는 멤버십을 만들 수 없습니다. 만료 판정은 반드시 서버 시간 기준으로.

🧩
규칙 안 get() 은 비용·복잡도가 있다

규칙에서 다른 문서를 get()으로 참조하면 읽기 1회가 더 들고 규칙이 복잡해집니다. 트래픽이 많거나 입장 로직이 더 정교해야 한다면, 멤버십 생성을 Cloud Function으로 옮기는 편이 깔끔합니다(섹션 5의 3계층). 함수 안에서 코드 존재·만료를 검사하고, 클라이언트의 memberships 직접 쓰기는 규칙으로 아예 막습니다.

9

한 장 요약

입장 코드 2주 만료 설계의 전체 그림입니다.

  1. 코드는 문서 ID로. joinCodes/ABCDEF — get은 누구나, list는 교사만. 긁어모으기 차단.
  2. 만료는 expiresAt으로. 발급 때 now + 14일을 박아 두고, 판정은 expiresAt < now 한 번.
  3. 입장은 3단계. 코드 존재 → 만료 안 됨 → 그때만 멤버십 생성.
  4. 검증은 서버에서. 클라이언트는 안내용, 보안 규칙·Cloud Function이 강제. 만료는 request.time 기준.
  5. 만료엔 재발급을 한 쌍으로. 자동 만료로 보안, 원클릭 재발급으로 잠김 복구. 코드 교체 방식이 더 안전.
  6. 코드 만료 ≠ 학생 추방. 소속은 memberships/{uid}_{classId}에 별도 저장 → 만료·기기변경과 무관하게 유지.
코드는 14일짜리 입장권, 멤버십은 평생 정기권. 만료는 새 손님만 막고, 이미 들어온 학생은 uid라는 신분증으로 어느 기기에서든 다시 들어온다.
