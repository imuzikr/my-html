GitHub 화면을 정확히 이해하는 한 장

GitHub는 내 폴더가 아니라 Git 저장소를 보여준다

GitHub의 파일 목록은 내 컴퓨터 작업 디렉토리를 실시간으로 비추는 창이 아닙니다. GitHub 서버에 push된 저장소에서 특정 커밋이 가리키는 tree와 blob을 읽어, 웹에서 폴더처럼 보이도록 렌더링한 화면입니다.

로컬 파일 → commit으로 기록 → push로 서버 복사본에 전달 → GitHub가 렌더링
github.com/team/project/tree/main

project

branch: main latest commit: a1b2c3d server-side repository
src tree 객체에서 온 디렉토리 항목
README.md blob 객체의 파일 내용
package.json 선택 커밋 기준
화면은 폴더 탐색기처럼 보이지만, 실제로는 선택된 커밋의 tree를 사람이 읽기 좋은 파일 목록으로 펼친 결과입니다.

한눈에 보는 흐름

1

작업 디렉토리

내 컴퓨터 폴더에 실제 파일이 펼쳐져 있습니다. 수정 중인 파일, 아직 commit하지 않은 파일도 여기에 있을 수 있습니다.

2

commit으로 기록

Git은 그 시점의 파일 스냅샷과 이력을 객체 데이터베이스에 저장합니다. 이때 commit, tree, blob이 연결됩니다.

3

로컬 .git 저장소

로컬 .git은 내 컴퓨터의 Git 저장소입니다. GitHub가 직접 이 폴더를 읽는 것은 아닙니다.

4

push로 서버에 복사

commit을 push하면 GitHub 서버의 저장소 복사본에 이력이 전달됩니다. commit만 하고 push하지 않으면 GitHub 화면은 바뀌지 않습니다.

5

GitHub가 렌더링

GitHub는 기본 브랜치의 최신 commit 같은 선택된 commit을 기준으로 파일 구조와 이력을 웹 화면에 보여줍니다.

Git 내부 객체 3종 세트

C

commit

어느 시점의 기록입니다. 작성자, 메시지, 부모 commit, 그리고 그 시점의 최상위 tree를 가리킵니다.

T

tree

디렉토리 구조입니다. 파일 이름, 하위 디렉토리, 각 항목이 가리키는 blob이나 tree 정보를 담습니다.

B

blob

파일 내용입니다. 이름이나 위치가 아니라 내용 자체가 저장되며, tree가 이 blob에 파일 이름을 붙여 연결합니다.

조금 부정확한 표현

GitHub는 내 컴퓨터의 `.git` 폴더를 렌더링한다.

왜 아쉬울까?
  • GitHub는 로컬 디스크에 접근하지 않습니다.
  • push되지 않은 commit은 GitHub에 없습니다.
  • working directory의 미커밋 변경도 보이지 않습니다.
헷갈리는 이유
  • 화면이 폴더 탐색기처럼 생겼습니다.
  • 로컬 `.git`과 GitHub 저장소가 같은 이력을 공유할 수 있습니다.

더 정확한 표현

GitHub는 서버에 저장된 Git 저장소의 객체 데이터베이스를 읽고, 선택된 commit의 tree와 blob을 파일 구조처럼 렌더링한다.

핵심 조건
  • GitHub 서버에 push되어 있어야 합니다.
  • 브랜치나 태그가 어느 commit을 가리키는지가 중요합니다.
보이는 것
  • 선택 commit의 파일 스냅샷
  • commit 이력과 비교 결과
  • tree와 blob을 해석한 웹 UI

비유: 필름 원본과 갤러리 전시

로컬 .git은 내 손에 있는 필름 원본에 가깝습니다. push는 그 필름의 복사본을 GitHub 서버에 맡기는 일이고, GitHub 웹 화면은 서버에 보관된 필름을 현상해 갤러리에 전시한 결과입니다.

→

한 문장으로 정리하면, GitHub는 내 작업 폴더를 그대로 보여주는 서비스가 아니라, push된 서버 측 Git 저장소에서 선택된 commit의 tree와 blob을 읽어 파일 목록과 이력으로 렌더링하는 서비스입니다. 그래서 GitHub에 보이게 하려면 파일 수정뿐 아니라 commit과 push까지 이어져야 합니다.
