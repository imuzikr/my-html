커밋은 작업 일지의 한 줄, 브랜치는 원본을 건드리지 않는 복사본. 이 두 가지만 알면 세 가지 합치기 방법이 명확하게 보입니다.

1

Git이란 무엇인가요?

Git은 파일의 변경 이력을 추적하고 관리하는 버전 관리 시스템(Version Control System)입니다. 코드뿐만 아니라 어떤 텍스트 파일이든 "언제, 누가, 무엇을 바꿨는지"를 기록해 두고, 필요하면 과거 어느 시점으로도 되돌릴 수 있습니다.

마치 문서 편집기의 실행 취소(Ctrl+Z)가 무한히 저장되는 것과 비슷하지만, Git은 여러 사람이 동시에 같은 프로젝트를 작업할 때도 서로의 변경 내용이 충돌 없이 합쳐질 수 있도록 도와준다는 점에서 훨씬 강력합니다. 오늘날 소프트웨어 개발 현장에서 사실상 표준으로 쓰이는 협업 도구입니다.

🕐
변경 이력 추적
누가 언제 무엇을 바꿨는지 모든 기록이 남습니다
↩️
되돌리기
실수해도 과거 어느 시점으로든 복구할 수 있습니다
🤝
협업
여러 사람이 동시에 작업해도 변경 내용을 안전하게 합칩니다
이 글의 핵심

Merge는 기록을 있는 그대로 보존하고, Rebase는 한 줄로 깔끔하게 정리하며, Squash는 자잘한 커밋들을 하나로 묶습니다. 셋 모두 브랜치를 합치는 방법이지만, 히스토리를 어떻게 남길지에 따라 선택이 달라집니다.

2

들어가기 전에 — 커밋과 브랜치가 뭔가요?

본격적인 이야기를 하기 전에, 두 단어만 알고 가면 됩니다.

커밋(Commit)

커밋은 작업하다가 그때그때 찍어 두는 사진 한 장입니다. "여기까지 했다"는 저장 지점이자, 언제·누가·무엇을 바꿨는지가 함께 기록되는 작업 일지의 한 줄이라고 생각하면 됩니다. 아래 그림에서 A, B, C 같은 알파벳 하나하나가 바로 커밋입니다.

브랜치(Branch)

브랜치는 단어 그대로 나뭇가지입니다.

📄
비유

여러 명이 함께 쓰는 공식 문서 원본이 하나 있다고 해봅시다. 원본에 직접 손대면 위험하니, 나는 복사본을 떠서 마음껏 고쳐 봅니다. 잘 안 되면 복사본만 버리면 되고, 잘 되면 원본에 반영합니다. 그 공식 문서 원본에 해당하는 큰 줄기가 main 브랜치이고, 내가 따로 떠낸 복사본이 feature 브랜치입니다.

가지를 치고 나가서(분기) 따로 작업하다가, 다 되면 다시 줄기로 돌아와 합치는 것 — 이 "합치는 방법"이 바로 오늘의 주제인 Merge, Rebase, Squash 세 가지입니다.

B 커밋에서 feature 브랜치가 갈라져 나온 모습. D, E는 feature에서만 작업한 커밋입니다.

3

Merge — 기록을 있는 그대로 합친다

가장 기본적인 방법입니다. 두 브랜치의 작업 내용을 합치면서 "병합했다"는 기록이 담긴 커밋(M)이 하나 새로 생깁니다. feature 브랜치에서 작업한 D, E가 그대로 보존되고, 언제 어디서 갈라져 언제 합쳤는지 전체 흐름이 히스토리에 남습니다.

M(병합 커밋)이 새로 생기며 두 가지가 하나로 모입니다. C, D는 그대로 보존됩니다.

📚
비유

두 권의 작업 일지를 시간 순서 그대로 둔 채, 마지막에 "이날 둘을 합쳤음"이라는 메모 한 장을 끼워 넣는 것. 기록이 사실 그대로 남는 대신, 협업하는 사람이 많아지면 히스토리가 여러 갈래의 그물처럼 보여 다소 복잡해질 수 있습니다.

터미널 명령어
git checkout main
git merge feature
4

Rebase — 커밋들을 줄기 끝에 옮겨 붙인다

feature 브랜치의 커밋들을 main 브랜치 끝에 옮겨 붙이는 방식입니다. 마치 처음부터 main에서 순서대로 작업한 것처럼 히스토리가 한 줄로 깔끔해집니다. 대신 "갈라져서 합쳤다"는 기록은 사라집니다.

C', D'는 C, D와 내용은 같지만 새로 생성된 별개의 커밋입니다. 히스토리가 한 줄로 정리됩니다.

⚠️
C'와 D' — 복사본이지 원본이 아닙니다

알파벳 뒤에 붙은 작은따옴표(')가 중요합니다. Rebase는 기존 커밋을 그대로 옮기는 게 아니라, 내용은 같지만 새로 만들어진 커밋으로 교체합니다. 사진을 옮겨 붙인 게 아니라, 같은 장면을 다시 찍은 셈입니다.

✏️
비유

feature에서 한 작업을 "원래 main에서 처음부터 했던 것처럼" 깨끗한 종이에 다시 베껴 쓰는 것. 결과물의 흐름이 한 줄로 정리되어 나중에 읽기 편합니다.

터미널 명령어
git checkout feature
git rebase main
# 이후 main으로 이동해서 fast-forward merge
git checkout main
git merge feature
5

Squash — 여러 커밋을 하나로 눌러 합친다

feature 브랜치의 여러 커밋을 하나로 합쳐서 main에 붙이는 방식입니다. feature에서 시행착오를 거치며 커밋을 여러 번 했더라도, main에는 깔끔하게 하나의 커밋으로만 남습니다.

C, D, E 세 커밋이 F 하나로 합쳐집니다. 시행착오의 흔적은 사라지고 결과만 남습니다.

🗜️
비유

여러 장의 사진을 한 장으로 합쳐 붙이는 것. "오타 수정 → 다시 수정 → 또 수정" 같은 자잘한 과정을 감추고, "이 기능 완성"이라는 결과 한 줄만 남기고 싶을 때 좋습니다.

터미널 명령어
git checkout main
git merge --squash feature
git commit -m "기능 완성: 로그인 폼 추가"
6

언제 뭘 쓰나요? — 세 방법 한눈에 비교

팀마다 선호하는 방식이 다르고 정해진 정답은 없습니다. 다만 아래 기준으로 고르면 대부분의 상황에서 적합한 선택이 됩니다.

방법 히스토리 형태 병합 커밋 기존 커밋 보존 주로 쓰는 때
Merge 그물 모양 (분기 그대로) 생성됨 보존 여러 명이 공유하는 브랜치를 합칠 때
Rebase 한 줄 (깔끔) 없음 새로 생성 혼자 쓰는 브랜치를 main 위에 정렬할 때
Squash 한 줄 (요약) 하나로 합쳐짐 사라짐 자잘한 커밋들을 정리해 PR 단위로 묶을 때
혼자 작업하는 브랜치에서는 Rebase나 Squash로 정리하고, 여러 명이 공유하는 브랜치에서는 Merge를 쓰는 경우가 많습니다.
7

초보자가 꼭 알아 둘 두 가지

⚠️
공유 브랜치에 Rebase·Squash 금지

이미 다른 사람과 공유한(원격에 올린) 브랜치는 Rebase나 Squash로 함부로 건드리지 마세요. 이 두 방식은 기존 커밋을 없애고 새 커밋으로 바꿔치기하는 작업입니다. 그래서 그 브랜치를 함께 쓰던 동료의 기록과 어긋나 충돌이나 꼬임이 생길 수 있습니다. "히스토리를 다시 쓰는 작업은 나 혼자 쓰는 브랜치에서만"이 안전한 원칙입니다.

💡
충돌(Conflict)은 잘못이 아닙니다

합치는 과정에서 "충돌(conflict)"이 나는 건 잘못이 아닙니다. 양쪽이 같은 부분을 동시에 고쳤을 때, Git이 어느 쪽을 택할지 몰라 사람에게 물어보는 자연스러운 과정입니다. 당황하지 말고 어느 내용을 남길지 직접 선택한 후 다시 커밋하면 됩니다.

이 두 가지만 기억해도 Git을 처음 쓰는 분이 흔히 겪는 실수의 절반은 피할 수 있습니다.

정리 — 세 가지 비유로 기억하세요

Merge는 두 작업 일지를 나란히 두고 마지막에 "합쳤음" 메모 한 장을 끼워 넣는 것. 기록이 사실 그대로 남습니다.

Rebase는 feature에서 한 작업을 원래 main에서 처음부터 했던 것처럼 깨끗한 종이에 다시 베껴 쓰는 것. 한 줄로 정돈된 히스토리를 만듭니다.

Squash는 여러 장의 사진을 한 장으로 합쳐 붙이는 것. 시행착오는 지우고 결과 하나만 남깁니다.

합치는 방법 중 가장 좋은 것은 없습니다.
어떤 히스토리를 남길지에 따라 고르면 됩니다.
Git 브랜치 합치기 세 가지 방법을 비유와 시각 자료로 정리한 글입니다. — 2026-05-31
