GitHub Security

Secret Scanning,
무엇이 자동으로 작동하는가

Public이냐 Private이냐에 따라 자동 여부가 완전히 달라집니다. 그리고 "push 시 검사"와 "push 차단"은 다른 기능입니다. 세 가지 기능을 정확히 구분해 봅니다.

Secret Scanning Push Protection GitHub Advanced Security 보안 자동화
핵심 요약

Public 저장소는 Secret Scanning과 Push Protection이 무료·자동으로 켜져 있습니다. Private/Internal 저장소는 GitHub Advanced Security(또는 Secret Protection) 라이선스가 있어야 하고 직접 활성화해야 합니다. 그리고 "push 시 스캔"과 "push 차단(Push Protection)"은 별개 기능이므로 정확히 구분해야 합니다.

1

Secret Scanning이란 무엇인가

GitHub의 Secret Scanning은 코드에 실수로 포함된 API 키, 토큰, 비밀번호 같은 민감한 값을 자동으로 탐지하는 기능입니다. 구글 API Key, AWS Access Key, Stripe 시크릿 키 등 수백 가지의 알려진 패턴을 인식합니다.

🔍
Partner Secret Scanning
시크릿이 발견되면 해당 토큰을 발급한 서비스 제공자(파트너)에게 자동으로 알립니다. 구글, AWS 등이 파트너로 참여합니다.
🚧
Push Protection
시크릿이 포함된 커밋의 push 자체를 차단합니다. 스캔 후 알리는 것이 아니라 업로드 전에 막는다는 점이 다릅니다.
📬
Alert & Notification
시크릿이 발견되면 저장소 관리자와 커밋 작성자에게 메일과 GitHub 알림을 보냅니다.
🛡️
두 기능의 차이

Secret Scanning은 이미 올라간 택배를 X-ray로 검사해서 위험물을 신고하는 것이고, Push Protection은 위험물을 담은 택배가 접수 창구에서 아예 반려되도록 막는 것입니다. 목적은 같지만 개입 시점이 다릅니다.

2

Public 저장소 — 자동, 무료, 항상 켜져 있음

공개 저장소에서는 두 기능 모두 자동으로, 무료로, 항상 활성화되어 있습니다. 별도 설정이나 라이선스가 필요하지 않습니다.

✅
Public 저장소 기본 동작

Partner Secret Scanning: push가 일어날 때마다 커밋 내용을 스캔하고, 알려진 시크릿 패턴이 발견되면 해당 서비스 파트너에게 자동으로 통보합니다. 사용자가 끌 수 없는 기본 동작입니다.

Push Protection: 시크릿이 포함된 커밋은 push 자체가 거부됩니다. 2024년부터 Public 저장소에 기본 활성화되었습니다.

실제 경험한 "Secrets detected" 경고 메일과 push 차단이 바로 이 기본 동작의 결과입니다. GitHub가 AIza로 시작하는 구글 API Key 패턴을 감지하는 순간 자동으로 작동한 것입니다.

3

Private / Internal 저장소 — 라이선스 필요

비공개 저장소에서 Secret Scanning을 사용하려면 GitHub Advanced Security(GHAS) 또는 별도 GitHub Secret Protection 라이선스가 필요합니다. 자동이 아니며 직접 켜야 합니다.

⚠️
Private 저장소 주의사항

비공개 저장소에 API 키가 포함된 코드를 push해도 라이선스 없이는 GitHub가 자동으로 감지하거나 차단하지 않습니다. 조직 정책으로 일괄 활성화할 수 있지만, 이 역시 라이선스가 전제입니다.

4

Push Protection — "스캔"과 "차단"은 다르다

"push할 때 자동으로 검사한다"고 하면 두 가지 의미가 섞일 수 있습니다. Secret Scanning은 push 이후에 커밋 내용을 분석하는 것이고, Push Protection은 push가 일어나는 시점에 개입해서 아예 업로드를 막는 것입니다.

📥
Secret Scanning (after push)
push가 완료된 후 저장소를 스캔합니다. 시크릿이 발견되면 알림을 보내지만, 코드는 이미 GitHub에 올라간 상태입니다.
🚧
Push Protection (before push)
push 요청이 들어오는 시점에 커밋을 검사합니다. 시크릿이 감지되면 push가 거부되어 코드가 GitHub에 도달하지 않습니다.

Push Protection이 차단하면 터미널에 오류 메시지가 나타나고, 사용자는 시크릿을 제거하거나 (위험을 인지한 채) 우회를 명시적으로 선택해야만 push를 재시도할 수 있습니다.

Push Protection은 "이미 들어온 것을 찾는" 것이 아니라 "들어오지 못하게 막는" 것입니다. 가장 효과적인 예방 수단입니다.

💡
개인 개발자에게 실질적인 의미

개인 프로젝트나 학습용 저장소는 대부분 Public입니다. 이 경우 별도 설정 없이도 API 키 노출이 자동으로 감지·차단됩니다. 다만 Private으로 전환하는 순간 이 보호막이 사라지므로, Private 저장소에서는 .env 파일과 .gitignore 관리가 더욱 중요합니다.

정리하면

  • Public 저장소: Secret Scanning + Push Protection 모두 무료·자동. 사용자가 끌 수 없는 기본 보호입니다.
  • Private/Internal 저장소: GitHub Advanced Security(GHAS) 또는 Secret Protection 라이선스가 있어야 하고, 설정에서 직접 켜야 합니다.
  • Secret Scanning은 push 완료 후 분석해서 알리는 것이고, Push Protection은 push 요청 시점에 차단하는 별개 기능입니다.
  • 노출된 시크릿은 코드에서 지운다고 해결되지 않습니다. 반드시 발급처에서 폐기(revoke)하고 재발급(rotate)해야 합니다.
  • 근본적인 예방책은 시크릿을 코드에 직접 쓰지 않고 .env 파일 + .gitignore 패턴을 쓰는 것입니다.

GitHub 공식 문서(Secret scanning, Push protection)를 바탕으로 작성한 개념 정리입니다.
