Ouroboros가 추구하는 방향성

Ouroboros의 핵심 방향은 AI에게 더 빨리 코드를 쓰게 하는 것이 아니라, 코드가 시작되기 전에 사람이 아직 정하지 않은 목표, 제약, 성공 기준, 개념 구조를 드러내는 데 있다. 한마디로 프롬프트 작성 보조가 아니라, 명세가 생길 때까지 질문하고 검증하는 개발 워크플로우를 지향한다.

README 영역 핵심 내용 문서에서의 의미
도입부 “프롬프트를 멈추고 명세를 시작하라”는 선언 Ouroboros가 프롬프트 작성 도구가 아니라 명세 생성 워크플로우임을 밝힌다.
Wonder에서 온톨로지로 소크라테스식 질문과 온톨로지 분석 “어떻게 하지?”보다 “이게 정확히 뭐지?”를 먼저 묻는 철학적 기반이다.
빠른 시작 설치, 런타임 감지, ooo interview 흐름 철학을 실제 CLI와 AI 코딩 런타임에서 바로 실행할 수 있게 만든다.
순환 구조 Interview -> Seed -> Execute -> Evaluate -> Evolve 한 번의 생성이 아니라 평가 결과를 다음 명세에 반영하는 진화 구조를 설명한다.
수치 기준 모호성 0.2 이하, 온톨로지 유사도 0.95 이상 작업 시작과 종료를 감이 아니라 게이트로 판단한다.
명령어와 에이전트 ooo seed, ooo run, ooo evaluate, 아홉 개의 사고 질문, 정의, 실행, 검증을 역할별로 나누어 품질을 높인다.
내부 구조와 TUI 라우팅, 실행, 평가, 진화, 관측성, 영속성, 런타임 추상화 README가 단순 컨셉 설명이 아니라 실제 시스템 설계를 함께 보여준다.

Problem

Ouroboros가 해결하려는 진짜 문제

AI가 부족해서가 아니라, 우리가 덜 정해서 실패한다
많은 AI 코딩 실패는 코드 생성 이후가 아니라 코드 생성 이전에 발생한다. 목표가 흐릿하고, 제약이 빠져 있고, 성공 기준이 측정되지 않으면 모델은 그 빈칸을 그럴듯한 추정으로 메운다. Ouroboros는 이 추정을 줄이는 방향으로 설계된다.

기존 AI 코딩 흐름

  • 아이디어를 바로 프롬프트로 던진다.
  • AI가 빠르게 구현안을 만든다.
  • 뒤늦게 “원하던 게 이게 아니었다”는 사실을 발견한다.
  • 재작업은 코드 문제가 아니라 요구 정의 문제로 돌아간다.

Ouroboros가 원하는 흐름

  • 소크라테스식 질문으로 숨은 가정을 드러낸다.
  • 도메인의 핵심 개념과 관계를 먼저 정리한다.
  • 모호성 점수가 기준 아래로 내려갈 때까지 명세를 다듬는다.
  • 실행 결과를 평가하고 다음 세대 명세에 반영한다.

Criteria

좋은 명세를 감이 아니라 수치와 게이트로 판정한다

Ouroboros의 중요한 특징은 “충분히 물어봤다”를 분위기로 판단하지 않는다는 점이다. README는 모호성을 1 - 가중 명확도로 계산하고, 그 값이 0.2 이하일 때 Seed를 만들 수 있다고 설명한다. 또한 진화 루프는 온톨로지 유사도 0.95 이상에서 수렴한다고 본다.

판정 대상 질문 기준 의미
목표 명확도 무엇을 만들려는지 구체적인가? Greenfield 40%, Brownfield 35% 아이디어가 실행 가능한 목적문으로 바뀌었는지 본다.
제약 명확도 시간, 기술, 운영, 사용자 제약이 정의됐는가? Greenfield 30%, Brownfield 25% AI가 임의로 빈칸을 채우지 않도록 경계를 만든다.
성공 기준 완료 여부를 측정할 수 있는가? Greenfield 30%, Brownfield 25% “작동한다”보다 구체적인 검증 조건을 요구한다.
컨텍스트 명확도 기존 코드베이스와 환경을 이해했는가? Brownfield 15% 이미 존재하는 구조를 무시한 새 구현을 막는다.
온톨로지 수렴 다음 세대에서도 같은 개념 구조가 유지되는가? Similarity 0.95 이상 더 질문해도 구조가 크게 바뀌지 않을 때 멈춘다.

Agents

아홉 개의 사고는 역할 분리가 아니라 판단의 안전장치다

필요할 때만 다른 사고 방식을 로드한다
Ouroboros는 모든 역할을 처음부터 켜두는 방식이 아니라, 질문이 필요한 순간에는 Interviewer, 본질을 구분해야 할 때는 Ontologist, 검증이 필요할 때는 Evaluator처럼 사고 모드를 분리한다.
질문

Socratic Interviewer

사용자가 당연하다고 넘긴 가정을 묻는다. 이 역할은 만들지 않고, 질문으로 문제의 모양을 선명하게 한다.

정의

Ontologist

증상과 본질을 구분하고, 도메인의 핵심 개념이 무엇인지 정리한다.

검증

Evaluator

기계적 검증, 의미적 검증, 다중 모델 합의를 거쳐 “맞는 것을 만들었는가”를 확인한다.

Concrete Direction

이 레포가 앞으로 밀고 가려는 제품 방향

프롬프트 작성 보조에서 명세 생성 엔진으로

사용자의 말 한 줄을 더 좋은 프롬프트로 바꾸는 수준을 넘어, 질문과 답변을 누적해 실행 가능한 Seed로 압축한다.

일회성 생성에서 세대 기반 진화로

한 번 만들고 끝나는 결과물이 아니라, 평가 결과가 다음 입력으로 돌아가며 점점 더 명확한 구조로 수렴하는 흐름을 만든다.

단일 모델 의존에서 검증 게이트로

모델 하나의 답을 그대로 믿지 않고, 비용이 낮은 기계적 확인부터 의미 평가와 합의까지 단계적으로 올린다.

런타임별 도구에서 에이전트 공통 워크플로우로

Claude Code, Codex CLI, Kiro CLI 같은 환경 위에 명세 우선 절차를 얹어, 특정 IDE보다 절차 자체를 중심에 둔다.

핵심 요약

Ouroboros는 AI 코딩의 속도보다 결정 품질을 다룬다

이 레포의 방향성은 “AI가 코드를 잘 쓰게 하자”보다 한 단계 앞에 있다. 좋은 코드는 좋은 결정에서 나오고, 좋은 결정은 명확한 목표, 제약, 성공 기준, 개념 구조에서 나온다는 관점이다. 그래서 Ouroboros는 인터뷰, Seed, 실행, 평가, 진화라는 순환을 통해 사람이 덜 정한 부분을 드러내고, 안정된 명세가 생길 때까지 작업을 늦춘다.

  • 철학: 구현보다 정의가 먼저다.
  • 방법: 소크라테스식 질문과 온톨로지 분석으로 숨은 가정을 찾는다.
  • 구조: Interview -> Seed -> Execute -> Evaluate -> Evolve의 순환을 따른다.
  • 판정: 모호성 0.2 이하, 유사도 0.95 이상 같은 게이트로 멈출 시점을 정한다.
  • 제품성: 여러 AI 코딩 런타임 위에서 재사용 가능한 명세 우선 워크플로우가 되려 한다.
