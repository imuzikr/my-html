Firebase DB 연결 시
6단계 심층 방어 전략

단 하나의 강력한 자물쇠보다, 성격이 다른 여러 개의 자물쇠를 직렬로 두는 편이 안전합니다. 각 레이어가 막는 위협의 종류와 동작 방식을 완전히 해설합니다.

2026-05-23 Firebase · Vercel · Security

심층 방어(Defense in Depth)란?

심층 방어는 보안 업계에서 널리 쓰이는 원칙으로, 하나의 강력한 레이어에 의존하는 대신 서로 다른 성격의 여러 레이어를 겹쳐 두는 전략입니다. 공격자가 한 레이어를 뚫어도 다음 레이어에서 막히며, 각 레이어가 잡는 위협의 종류가 달라 빈틈이 줄어듭니다.

Firebase DB를 웹 앱에 연결할 때는 특히 이 원칙이 중요합니다. Firebase SDK 설정 값은 클라이언트 코드에 포함되어 누구나 볼 수 있기 때문입니다. 따라서 SDK 키가 노출되더라도 각 레이어가 악용을 막을 수 있어야 합니다.

전체 요청 흐름 — 6개 레이어 통과 구조
ℹ️
레이어별 위치가 다릅니다 일직선 흐름처럼 보여도 실제로는 서로 다른 인프라에서 강제됩니다. 1·2번은 Google 인프라, 3번은 Firebase Auth 서버, 4·5번은 Cloud Functions, 6번은 Firebase DB 엔진에서 각각 동작합니다. 하나가 뚫려도 다음 레이어가 별도 시스템에서 재검사합니다.
1

리퍼러(Referrer) 검사

HTTP Referer 헤더로 요청 출처 도메인 확인

보안 평가
★★★★★
가장 약한 레이어 · 1차 필터

브라우저가 요청을 보낼 때 HTTP Referer 헤더에 "이 요청은 어느 페이지에서 왔습니다"라는 정보를 함께 전송합니다. Google Cloud Console의 API 키 설정에서 승인된 도메인 목록을 등록하면, 목록 외의 도메인에서 온 요청은 Firebase가 거부합니다.

클럽 입구에서 "어느 식당에서 오셨어요?"라고 묻고 말만 듣고 통과시키는 방식입니다. 신분증 확인이 아닌, 자기 신고에 의존하는 구조입니다.

막을 수 있는 것
  • ✓다른 도메인에서 실수로 우리 API 키를 가져다 쓰는 경우
  • ✓브라우저 기반 CORS 제한과 함께 쓰면 1차 필터 역할
막지 못하는 것
  • ✗curl / Postman 등으로 헤더를 위조한 요청
  • ✗프라이버시 확장으로 Referer를 숨긴 정상 사용자 오탐
  • ✗진지한 공격자 — Referer 위조는 1초면 가능
⚠️
이 레이어만 믿지 마세요 Referer 검사는 "방어의 첫 문"이지 "튼튼한 자물쇠"가 아닙니다. 공격자가 아닌 일반 자동화 도구도 손쉽게 통과할 수 있습니다. 반드시 다음 레이어와 함께 사용해야 합니다.
2

App Check

reCAPTCHA Enterprise · 진짜 앱에서 온 요청인지 암호학적으로 검증

보안 평가
★★★★★
봇·자동화 공격에 매우 효과적

Firebase App Check는 Google의 reCAPTCHA Enterprise를 활용해 요청이 진짜 웹 앱에서 왔는지 검증합니다. 마우스 움직임, 브라우저 환경, 행동 패턴 등 수십 가지 신호를 분석해 사람인지 봇인지 판별하고, 통과하면 짧은 수명의 토큰을 발급합니다.

1번과 결정적으로 다른 점은 암호학적 서명이 된 토큰을 사용한다는 것입니다. 단순히 헤더 값을 바꾸는 것으로는 위조할 수 없고, Google 서버에서 발급된 유효한 토큰이 없으면 Firebase가 요청을 거부합니다.

막을 수 있는 것
  • ✓자동화 봇의 대량 API 호출
  • ✓스크래핑 · 크리덴셜 스터핑
  • ✓직접 HTTP 요청으로 Firebase 접근 시도
한계
  • △Puppeteer 등 실제 브라우저 자동화는 일부 우회 가능
  • △reCAPTCHA 점수 임계값 조정이 필요할 수 있음
3

Google 로그인 (Firebase Authentication)

신원 확인을 Google에 위임 · JWT 토큰으로 사용자 식별

보안 평가
★★★★★
신원 위조 방어 · 매우 강력

사용자가 Google 계정으로 로그인하면 Firebase는 ID 토큰(JWT)을 발급합니다. 이 토큰에는 사용자의 고유 ID, 이메일, 만료 시각 등이 담겨 있고, Google의 비공개 키로 서명되어 있어 위조가 사실상 불가능합니다.

비밀번호 저장, 2단계 인증, 비정상 로그인 감지 같은 어려운 보안 문제를 Google이 대신 처리합니다. 단, 인증(Authentication)과 인가(Authorization)는 다릅니다. 로그인 여부(인증)가 특정 데이터에 접근할 권한(인가)을 보장하지 않으며, 인가는 5·6번 레이어에서 처리합니다.

🔑
JWT 토큰의 핵심 특성 Header.Payload.Signature 세 부분으로 구성됩니다. Signature는 Google 서버의 비공개 키로 생성되며, 누구나 Google의 공개 키로 검증할 수 있지만 위조는 불가능합니다. 토큰 만료 후에는 자동으로 갱신되며, 서버 측 세션 저장소가 필요 없습니다.
4

사용량 제한 (Rate Limiting)

IP / 사용자 / 엔드포인트별 요청 빈도 제한

보안 평가
★★★★★
세밀한 정책 설계가 핵심

Firebase에는 강력한 내장 Rate Limiting이 없습니다. 보통 Cloud Functions 안에서 직접 구현하거나 Firestore/Redis 카운터를 활용합니다. "같은 사용자가 1분에 100번 이상 요청 → 차단", "같은 IP에서 시간당 쓰기 10번 초과 → 거부" 같은 규칙이 전형적인 예입니다.

⚡ 막는 위협 3가지
  • ①DoS 공격 — 대량 요청으로 서비스 다운 시도 차단
  • ②비용 폭탄 — Firebase는 사용량 과금, 악의적 호출로 청구서 폭발 방지
  • ③무차별 대입 — 비밀번호 · 쿠폰 코드 추측 공격 제한
⚠️ 설계 시 주의점
  • △IP 기반 제한은 NAT/VPN 환경에서 정상 사용자도 차단 가능
  • △사용자별 · IP별 · 엔드포인트별 정책을 분리 설계할 것
  • △Firestore 카운터 구현 시 트랜잭션으로 동시성 문제 처리 필요
typescript · Cloud Functions Rate Limiter 예시 
async function checkRateLimit(userId: string, action: string) {
  const key = `ratelimit/${userId}/${action}`;
  const ref = db.doc(key);

  return db.runTransaction(async tx => {
    const doc = await tx.get(ref);
    const now = Date.now();
    const data = doc.data() ?? { count: 0, windowStart: now };

    // 1분 창(window) 기반 카운터
    if (now - data.windowStart > 60_000) {
      tx.set(ref, { count: 1, windowStart: now });
      return true;
    }
    if (data.count >= 100) throw new Error("rate_limit_exceeded");
    tx.update(ref, { count: data.count + 1 });
    return true;
  });
}
5

Cloud Functions (서버 코드)

민감한 로직을 서버에 격리 · 클라이언트가 DB에 직접 쓰지 못하도록

보안 평가
★★★★★
비즈니스 로직 보호 · 핵심 레이어

클라이언트 코드는 브라우저 개발자 도구로 누구나 볼 수 있습니다. 따라서 민감한 비즈니스 로직, API 키, 트랜잭션 처리는 반드시 서버(Cloud Functions)에서 실행해야 합니다.

예를 들어 "포인트 차감 후 상품 구매" 로직을 클라이언트에서 두 번의 DB 호출로 처리하면, 공격자가 두 번째 호출(상품 지급)만 골라 실행하는 우회가 가능합니다. Cloud Functions에서 트랜잭션으로 묶으면 원천 차단됩니다.

🚨
클라이언트 측 검증은 UX용입니다 — 보안이 아닙니다 React나 Vue에서 아무리 철저하게 입력값을 검증해도, 공격자는 브라우저를 거치지 않고 직접 HTTP 요청을 보낼 수 있습니다. 입력값 검증은 반드시 Cloud Functions에서 다시 한번 수행해야 합니다.
서버에서 처리해야 할 것들
  • ✓결제 금액 계산 · 재고 차감 트랜잭션
  • ✓외부 결제 API 키 (Stripe, Toss 등)
  • ✓사용자 권한 승격 · 관리자 작업
  • ✓개인정보 처리 · 이메일 발송
서버 측 입력값 검증 예시
  • ✓가격이 음수가 아닌지 확인
  • ✓주문 수량이 재고를 초과하지 않는지
  • ✓문자열 길이 · 허용 문자 범위 검사
  • ✓요청 사용자 ID가 JWT와 일치하는지
6

Firebase 보안 규칙 (최후의 방어선)

Firestore · Storage · Realtime DB에 선언적 접근 제어 · Firebase 엔진이 강제

보안 평가
★★★★★
가장 중요 · 절대 생략 불가

앞의 다섯 레이어가 모두 뚫려도 이 레이어가 막아냅니다. Firebase DB에 직접 규칙을 선언하고, Firebase 인프라 자체가 모든 접근을 이 규칙으로 재검사합니다. 어떤 경로로 요청이 들어오든 예외 없이 적용됩니다.

단, 이 레이어가 가장 실수가 잦은 곳이기도 합니다. 개발 편의를 위해 allow read, write: if true로 열어두고 배포하면 전체 DB가 인터넷에 공개됩니다.

firestore.rules · 실전 예시 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // 게시글: 누구나 읽기, 작성자만 수정/삭제
    match /posts/{postId} {
      allow read: if true;
      allow create: if request.auth != null;
      allow update, delete: if
        request.auth != null
        && request.auth.uid == resource.data.uid;
    }

    // 사용자 프로필: 본인만 읽기·수정
    match /users/{userId} {
      allow read, write: if
        request.auth != null
        && request.auth.uid == userId;
    }

    // 어드민 컬렉션: 커스텀 클레임으로 검사
    match /admin/{doc} {
      allow read, write: if
        request.auth.token.admin == true;
    }
  }
}
🔥
흔한 실수 — 절대 하지 마세요 allow read, write: if true는 개발 편의를 위해 쓰는 임시 설정입니다. 이 상태로 배포하면 인터넷 상의 누구나 DB의 모든 데이터를 읽고 쓸 수 있습니다. 배포 전 반드시 Rules Playground에서 "비로그인 사용자가 남의 글을 수정할 수 있는가" 등의 시나리오를 테스트하세요.
✅ 배포 전 체크리스트
  • ✓Rules Playground에서 비인증 접근 차단 확인
  • ✓타인 문서 수정 불가 확인
  • ✓Firebase 에뮬레이터 단위 테스트 통과
  • ✓if true 규칙이 남아있지 않은지 전체 검색
🔍 테스트 시나리오 예시
  • ?비로그인 사용자가 게시글을 수정할 수 있는가?
  • ?로그인했지만 타인의 게시글을 삭제할 수 있는가?
  • ?일반 사용자가 어드민 컬렉션에 접근할 수 있는가?
  • ?작성자 본인은 자신의 글을 수정할 수 있는가?

Vercel 배포 환경에서 추가로 챙길 것

승인된 도메인 등록

Firebase Authentication 설정에서 Vercel 도메인을 반드시 추가해야 로그인이 동작합니다. PR 미리보기 URL(*.vercel.app)까지 허용할지 운영용만 등록할지 정책을 정하세요.

환경 변수 구분

Firebase SDK 키(NEXT_PUBLIC_)는 클라이언트에 노출되어도 괜찮습니다. 결제 API 키, 서비스 계정 키 같은 진짜 비밀은 NEXT_PUBLIC_ 없이 Vercel에 저장하고 서버 코드에서만 접근하세요.

⚠️
NEXT_PUBLIC_ 접두사 주의 Next.js에서 NEXT_PUBLIC_FIREBASE_SERVICE_ACCOUNT_KEY처럼 실수로 비밀 키에 이 접두사를 붙이면 브라우저 번들에 포함되어 전 세계에 공개됩니다. 비밀 키에는 절대 NEXT_PUBLIC_을 붙이지 마세요.
