Firestore 핵심 함수 4가지

doc · getDoc · setDoc · onSnapshot — 좌석 신청 앱 예시로 "주소표 만들기"부터 "읽기·쓰기·실시간 구독"까지 차근차근 풀어봅니다.

doc

문서를 가리키는 주소표(참조)를 만든다. 데이터는 아직 건드리지 않음.

getDoc

그 주소로 가서 내용을 한 번 읽어온다.

setDoc

그 주소에 데이터를 저장(쓰기)한다.

onSnapshot

그 문서를 계속 지켜보다가 바뀔 때마다 알려준다(실시간 구독).

한눈에 보는 전체 그림

이 네 함수는 역할이 분명히 나뉩니다. 핵심은 "주소를 만드는 단계"와 "그 주소로 실제 일을 하는 단계"가 따로라는 점이에요.

택배에 비유하면

doc은 주소가 적힌 송장을 만드는 일이에요. 송장을 만들었다고 물건이 오가는 건 아니죠. getDoc은 그 주소에서 물건을 가져오는 기사, setDoc은 그 주소로 물건을 보내는 기사예요. onSnapshot은 그 집 앞에 상주하며 변화를 알려주는 사람이고요.

실선 = 한 번 오가는 요청, 점선 = 계속 이어진 실시간 연결

doc 문서를 가리키는 주소표 만들기

doc은 Firestore의 어떤 문서를 가리키는 참조(reference)를 만드는 함수입니다. 중요한 점은, 이 순간에는 데이터를 읽지도 쓰지도 않는다는 거예요. 서버에 접속조차 하지 않습니다. 그냥 "이 문서를 가리키는 표지판" 하나를 손에 쥐는 것뿐이에요.

javascript
const ref = doc(db, "config", "main");

// db = 어느 데이터베이스, "config" = 컬렉션(폴더), "main" = 문서(파일)

매개변수의 의미

괄호 안의 값들은 "어느 문서를 가리킬지"를 알려주는 경로예요. Firestore는 폴더(컬렉션)와 파일(문서) 구조라서, 슬래시 경로처럼 단계를 나눠 적습니다.

  • db — 어느 데이터베이스인지
  • "config" — 컬렉션(폴더) 이름
  • "main" — 그 안의 문서(파일) 이름

그래서 위 코드는 config/main 문서를 가리키게 됩니다.

비유

doc(...)은 어떤 집의 주소를 적은 쪽지를 만드는 일이에요. 주소를 적었다고 그 집에 들어가거나 안을 본 건 아니죠. 주소만 손에 든 상태입니다.

왜 따로 떼어 놓았을까

읽기·쓰기·구독을 안 하고 "가리키기"만 분리해 둔 이유는, 같은 주소표 하나를 만들어 두고 여러 동작에 재사용하기 위해서예요. 실제 앱에서는 이 doc을 작은 함수로 감싸서 반복을 줄입니다.

javascript
// 자주 쓰는 주소를 함수로 묶어 단축 표현 만들기

const cfgRef = () => doc(db, "config", "main");



// 좌석·학생 문서도 같은 방식

const seatRef = (n) => doc(db, "weeks", weekId, "seats", String(n));

const stuRef  = (sid) => doc(db, "weeks", weekId, "students", sid);
자주 하는 실수 · 괄호 빠뜨리기

cfgRef는 함수입니다. 실제 주소표를 얻으려면 cfgRef()처럼 괄호를 붙여 호출해야 해요. 괄호 없이 setDoc(cfgRef, ...)로 쓰면 "함수 자체"를 넘기게 되어 오류가 납니다. 올바른 형태는 setDoc(cfgRef(), ...).

getDoc 그 주소에서 한 번 읽어오기

getDoc은 doc이 만든 주소표를 받아서, 그 문서의 내용을 딱 한 번 읽어오는 함수입니다. 여기서 비로소 서버와 통신이 일어나요.

javascript
const ref  = doc(db, "config", "main");  // ① 주소표 만들기 (통신 없음)

const snap = await getDoc(ref);          // ② 서버에서 실제로 읽어오기



if (snap.exists()) {

  console.log(snap.data());  // 예: { openOverride: false }

}

반환값 다루기 — snapshot

getDoc이 돌려주는 건 데이터 자체가 아니라 스냅샷(snapshot)이라는 포장지예요. 그 안에서 값을 꺼내려면 메서드를 씁니다.

  • snap.exists() — 그 문서가 실제로 있는지 확인(true/false)
  • snap.data() — 문서의 실제 내용을 꺼냄(없으면 undefined)
await가 붙는 이유

읽기는 서버를 다녀오는 일이라 시간이 걸려요. await는 "읽기가 끝날 때까지 기다렸다가 다음 줄로 가라"는 뜻입니다. 그래야 받아온 데이터를 다음 줄에서 안전하게 쓸 수 있어요.

핵심 차이

getDoc은 그 순간의 값을 한 번만 가져옵니다. 가져온 뒤 서버에서 값이 바뀌어도 다시 알려주지 않아요. "지금 상태가 궁금할 때 한 번 물어보는" 용도예요. 계속 지켜봐야 한다면 ④의 onSnapshot을 씁니다.

setDoc 그 주소에 데이터 저장하기

setDoc은 지정한 문서에 데이터를 저장(쓰기)하는 함수입니다. 좌석 신청 앱에서는 신청 상태나 좌석 상태를 저장할 때 쓰여요.

javascript
// config/main 문서에 openOverride: false 저장

await setDoc(cfgRef(), { openOverride: false }, { merge: true });

//            ↑ 저장 위치        ↑ 저장 내용              ↑ 저장 방식 옵션

가장 중요한 것 — merge 옵션

setDoc에서 제일 헷갈리는 게 merge 옵션이에요. 이게 있고 없고에 따라 동작이 크게 달라집니다.

형태동작비유
setDoc(ref, {a:1})
(merge 없음)		 문서를 통째로 덮어씀. 기존의 다른 값(b, c…)은 전부 사라짐 종이를 새 종이로 갈아 끼우기
setDoc(ref, {a:1}, {merge:true}) 기존 내용은 두고 지정한 값만 갱신·추가 기존 종이에서 한 줄만 고쳐 쓰기

비슷한 쓰기 함수와의 차이

  • setDoc — 이 문서를 이 내용으로 만들기(없으면 새로, 있으면 덮거나 병합)
  • updateDoc — 이미 있는 문서의 일부만 수정(문서가 없으면 오류)
  • addDoc — 문서 이름을 자동으로 정해 새로 추가

좌석 앱은 config/main처럼 이름이 정해진 문서를 다루고, 없으면 만들고 있으면 일부만 고치고 싶어서 setDoc + merge:true 조합을 씁니다.

보안 한 가지 · 쓰기 성공은 규칙이 정한다

setDoc은 누구나 SDK에서 꺼내 쓸 수 있는 공개 도구입니다. 따라서 이 함수로 무엇을 저장할 수 있는지는 클라이언트 코드가 아니라 서버의 보안 규칙이 결정해요. setDoc은 "저장 시도"일 뿐이고, 실제 저장 여부는 규칙이 허락해야 이뤄집니다.

onSnapshot 실시간으로 지켜보기

onSnapshot은 문서(또는 컬렉션)를 계속 구독합니다. 한 번 읽고 끝나는 getDoc과 달리, 그 데이터가 바뀔 때마다 자동으로 다시 알려줘요.

비유

유튜브 채널을 구독해 두면 새 영상이 올라올 때마다 알림이 오죠. onSnapshot도 똑같아요. 한 번 걸어두면, 그 문서가 바뀔 때마다 알림처럼 새 데이터가 자동으로 들어옵니다.

javascript
onSnapshot(cfgRef(), (snap) => {

  const c = snap.data() || {};       // 문서가 비어도 안전하게

  configOpenOverride =

    (typeof c.openOverride === "boolean") ? c.openOverride : null;

  refreshLockView();                  // 값에 맞춰 화면 다시 그리기

});

두 가지 인자

  • 지켜볼 대상 — cfgRef()가 가리키는 문서
  • 변화가 올 때 실행할 함수 — (snap) => { ... }. Firebase가 변화를 감지할 때마다 이 함수를 대신 실행하고, snap에 그 순간의 데이터를 담아 넣어줍니다.

(snap) => { ... } 모양을 화살표 함수라고 해요. "재료(snap)를 받아 중괄호 안의 일을 하는 작은 함수"라는 뜻입니다.

왜 좌석 앱에 꼭 필요할까

좌석 신청은 여러 학생이 동시에 같은 화면을 봅니다. 누군가 한 자리를 차지하면 다른 모든 학생 화면에서도 그 자리가 즉시 "신청완료"로 바뀌어야 하죠. onSnapshot이 좌석 문서들을 구독하고 있어서, 변화가 생기는 순간 모두의 화면이 새로고침 없이 갱신됩니다. 관리자가 신청을 열고 닫는 config/main 스위치도 같은 방식으로 전체에 실시간 전파돼요.

getDoc과 한 줄 비교

getDoc = "지금 값 한 번 알려줘"(스냅샷 1장). onSnapshot = "앞으로 바뀔 때마다 계속 알려줘"(연속 구독). 실시간 화면에는 onSnapshot, 한 번만 확인하면 될 때는 getDoc.

네 함수 한 장 요약

함수역할서버 통신반복성
doc주소표(참조) 만들기없음—
getDoc한 번 읽기1회일회성
setDoc저장(쓰기)1회일회성
onSnapshot실시간 구독지속 연결바뀔 때마다 반복

기억할 한 가지: doc은 "어디"를 정하고, 나머지 셋은 그 "어디"에 대해 각각 읽기·쓰기·지켜보기를 합니다.
