CORS는 왜 막고,
무엇으로 푸는가

막고 있는 진짜 주체는 동일 출처 정책(SOP)이며, CORS는 그 차단을 안전하게 푸는 약속이다. 미들웨어·라우터·개발 프록시까지 SVG 다이어그램으로 한 번에 이해한다.

대상: 백엔드를 처음 다루는 프론트엔드 개발자 · 2026-05-19

핵심 메시지
CORS는 차단하는 규칙이 아니라, 차단을 풀기 위한 약속이다.
막고 있는 진짜 주체는 동일 출처 정책(SOP)이며,
CORS는 서버가 응답 헤더를 통해 "이 출처는 읽어도 좋다"고 표시하면
브라우저가 그 표시를 보고 응답을 통과시키는 표준 메커니즘이다.

01

막는 것은 CORS가 아니라 동일 출처 정책이다

가장 먼저 바로잡아야 할 오해는 "CORS가 요청을 막는다"는 표현이다. 실제로 막고 있는 것은 동일 출처 정책(Same-Origin Policy, SOP)이라는, 브라우저에 내장된 더 근본적인 보안 정책이다. SOP는 "브라우저에서 실행되는 스크립트는 자신이 로드된 출처와 다른 출처의 응답을 자유롭게 읽을 수 없다"고 못 박는다. 악의적인 사이트가 사용자의 인증 쿠키를 이용해 다른 사이트의 데이터를 몰래 가져가는 것을 원천 차단하기 위함이다.

그림 1. SOP는 브라우저에 내장된 정책으로, 다른 출처 서버의 응답을 스크립트가 읽지 못하게 막는다

출처(Origin)란 정확히 무엇인가

출처는 스킴 + 호스트 + 포트 세 요소의 조합이다. 세 요소 중 하나라도 다르면 브라우저는 즉시 "다른 출처"로 판정한다. 로컬 개발에서 localhost:5173과 localhost:3000이 부딪히는 것도 포트 하나가 달라서다.

그림 2. 출처를 구성하는 세 요소. localhost:5173과 localhost:3000도 포트가 달라 다른 출처로 판정된다

구성 요소의미다른 출처가 되는 예
scheme통신 프로토콜http:// ↔ https://
host도메인 + 서브도메인example.com ↔ api.example.com
port접속 포트localhost:5173 ↔ localhost:3000
⚠ 흔한 오해 바로잡기

"CORS 때문에 막혔다"는 말은 일상적으로 통용되지만, 정확히는 SOP가 막고 있고, 그 제약을 풀어주는 약속이 CORS다. CORS(Cross-Origin Resource Sharing)는 "다른 출처 자원의 공유"라는 이름 그대로, 차단을 푸는 쪽의 표준이다.

02

서버가 "허용한다"고 말하는 방법: 응답 헤더

CORS의 핵심 아이디어는 단순하다. 서버가 응답을 보낼 때 "이 출처에서 오는 요청은 받아들이겠다"는 메시지를 함께 첨부하는 것이다. 이 메시지가 담기는 곳이 HTTP 응답 헤더(Response Header)다. 헤더는 본문(데이터) 앞에 붙는 메타정보로, 우편물의 봉투에 적힌 발신·수신 정보와 같은 역할을 한다.

브라우저는 응답이 도착하면 본문보다 먼저 헤더를 읽는다. Access-Control-Allow-Origin에 자신의 출처가 적혀 있으면 응답을 스크립트에 넘겨주고, 헤더가 없거나 다른 출처가 적혀 있으면 응답은 이미 도착했더라도 폐기된다.

그림 3. 브라우저는 응답 헤더의 Access-Control-Allow-Origin을 확인하고, 허용된 경우에만 응답을 스크립트에 전달한다

ℹ 핵심 정리

CORS 해결의 본질은 서버가 응답 헤더를 통해 신뢰를 명시적으로 표시하는 것이다. 클라이언트가 할 수 있는 일은 없으며, 결정권은 전적으로 서버 쪽에 있다. 응답이 이미 도착했더라도 헤더가 없으면 브라우저가 폐기한다.

03

단순 요청과 사전 요청(Preflight)

CORS 요청은 두 종류로 나뉜다. 브라우저는 위험할 수 있는 요청에 대해 본 요청을 보내기 전에 OPTIONS 메서드로 "보내도 되는가?"를 먼저 물어본다.

구분조건동작
단순 요청 GET·HEAD·POST 중 하나, CORS 안전 목록 헤더만, Content-Type이 text/plain · urlencoded · multipart 중 하나 본 요청을 곧바로 보내고 응답 헤더로 허용 여부 판단
사전 요청 (Preflight) PUT·DELETE, Content-Type: application/json, Authorization 헤더 등 위 조건을 벗어나는 모든 경우 OPTIONS로 먼저 허락 구한 뒤 본 요청 전송 (왕복 1회 추가)

그림 4. Preflight 흐름. PUT·DELETE·JSON 요청은 OPTIONS로 먼저 허락을 구하고, 허용 응답을 받은 뒤에야 본 요청을 보낸다

ℹ 왜 사전 요청이 필요한가

한 번의 본 요청만으로 데이터가 변경되거나 삭제되는 사고를 막기 위해서다. 위험할 수 있는 요청은 보내기 전에 허락을 먼저 받는다는 안전장치인 셈이다.

인증 쿠키가 있을 때 추가 제약

인증 쿠키를 함께 보내는 요청이라면 Access-Control-Allow-Credentials: true까지 필요하다. 이 경우 Allow-Origin을 와일드카드 *로 둘 수 없다는 제약이 따라온다. 반드시 특정 출처를 명시해야 한다.

04

누가 헤더를 붙이는가: 미들웨어·라우터·라우트 핸들러

CORS 헤더를 자동으로 챙겨주는 것은 백엔드 서버의 미들웨어(Middleware)라는 부품이다. 미들웨어를 이해하려면 함께 등장하는 라우터·라우트 핸들러까지 한 묶음으로 보아야 한다.

부품역할비유
미들웨어 요청이 최종 처리자에게 가기 전후에 끼어들어 공통 작업 수행 (CORS 헤더, 인증 검사, 로그 기록 등) 컨베이어 벨트 위 각 공정
라우터 들어온 요청의 URL과 메서드를 보고 어느 핸들러에 보낼지 분배하는 시스템 전화 교환대
라우트 핸들러 특정 경로·메서드 한 쌍에 대응해 실제 비즈니스 로직을 수행하고 응답을 만드는 함수 전화를 받아 업무를 처리하는 담당자

그림 5. 요청 처리 파이프라인. CORS 미들웨어는 라우터보다 앞단에 등록되어, 어느 경로로 가든 응답에 헤더가 일관되게 붙는다

✓ 실무 팁 (Node.js · Express)

직접 만들 필요 없이 cors npm 패키지를 설치해 한 줄로 등록하는 것이 표준이다.

JavaScript · Express 
const express = require('express');
const cors = require('cors');
const app = express();

// CORS 미들웨어를 라우터보다 앞에 등록
app.use(cors({
  origin: 'https://myapp.com',   // 허용할 출처 명시
  credentials: true             // 쿠키 포함 허용
}));

// 이후 라우터 등록
app.get('/api/users', (req, res) => {
  res.json({ users: [] });
});
⚠ 라우터와 핸들러 혼동 주의

라우터는 분배 시스템이고, 라우트 핸들러는 분배의 종착지다. 라우터는 다른 서버로 전달하는 게 아니라, 같은 서버 안에서 적절한 함수에 연결할 뿐이다. 실제 데이터 처리와 응답 생성은 핸들러가 직접 끝낸다.

05

개발용 프록시의 진짜 동작과 한계

개발 중에는 백엔드 코드를 한 줄도 짜지 않았는데도 외부 API를 잘 불러오는 경험을 한다. 흔히 "Vite 프록시가 CORS 헤더를 붙여서 돌려준다"고 설명하지만, 이는 정확하지 않다. 개발 서버 프록시는 헤더를 추가하지 않는다. 대신 요청을 동일 출처에서 받게 만들어 CORS 검사가 발생할 여지 자체를 없앤다.

핵심은 CORS는 브라우저↔서버 통신에만 적용되고, 서버↔서버 통신에는 적용되지 않는다는 점이다. 프록시는 이 비대칭을 활용해 브라우저 입장에서는 동일 출처로 통신하는 것처럼 만들어 준다.

그림 6. 프록시는 CORS 헤더를 추가하는 게 아니라, 요청을 동일 출처로 받아 CORS 검사 자체를 발생시키지 않는다

✗ 프록시의 결정적 한계

프록시는 개발용 서버가 띄워져 있을 때만 작동한다. 빌드 후 정적 파일로 배포하면 프록시 서버는 사라지고, 브라우저는 다시 외부 API에 직접 요청을 보내야 한다. 운영 환경에서는 자체 백엔드 서버에 CORS 미들웨어를 두거나, 백엔드가 외부 API를 대신 호출하는 구조로 바꿔야 한다.

이 한계를 이해하고 나면, 왜 많은 글이 "API 서버를 직접 만들라"고 권하는지가 분명해진다. 개발 중의 편의는 잠깐의 임시방편일 뿐이고, 배포된 앱이 외부 API를 안정적으로 다루려면 결국 자기 출처에서 응답해 줄 수 있는 서버가 있어야 하기 때문이다.

개념 총정리

동일 출처 정책 (SOP)
다른 출처의 응답을 스크립트가 읽지 못하게 막는 브라우저 기본 정책. 막는 진짜 주체.
CORS
SOP의 제약을 안전하게 푸는 표준. 차단하는 규칙이 아니라 차단을 여는 약속.
출처 (Origin)
스킴 + 호스트 + 포트의 조합. 셋 중 하나라도 달라지면 다른 출처.
사전 요청 (Preflight)
위험할 수 있는 요청 전에 OPTIONS로 미리 묻는 절차. 왕복 1회 추가.
응답 헤더
본문 앞에 붙는 메타정보. Access-Control-Allow-Origin이 CORS의 핵심 헤더.
미들웨어
요청·응답 사이에서 공통 작업을 수행. CORS 헤더를 모든 응답에 자동으로 붙임.
라우터
요청 URL·메서드를 보고 적절한 핸들러로 분배하는 시스템. 전화 교환대.
개발용 프록시
같은 출처로 요청을 받아 외부 서버에 대신 전달. 배포 시 사라지는 임시 방편.

막는 주체는 SOP, 푸는 약속은 CORS
서버는 응답 헤더로 신뢰를 표시하고, 위험한 요청엔 Preflight가 추가된다.
이 헤더를 일관되게 붙이는 건 미들웨어가 맡고,
개발 프록시는 헤더를 추가하는 게 아니라 CORS 검사 자체를 우회시키는 것이며
배포 시엔 사라지므로 운영 환경엔 자체 API 서버가 필요하다.
