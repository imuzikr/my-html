Family Daily English Automation

매일 아침 7시, 가족 생활영어가 자동으로 올라오는 웹사이트 만들기

Upstage Solar Pro 3가 생활영어 대화문을 만들고, Python이 HTML로 바꾸며, GitHub Actions가 GitHub Pages까지 배포합니다. 서버를 따로 운영하지 않아도 되는 작고 완결된 자동화 프로젝트입니다.

이 실습의 완성 조건

Actions를 한 번 수동 실행했을 때 daily/YYYY-MM-DD.html이 생기고, 메인 목록에 카드가 추가되며, 공개 Pages 주소에서 새 대화문을 바로 열 수 있으면 성공입니다.

01

전체 흐름부터 이해하기

자동화의 핵심은 “생성, 저장, 배포”를 한 워크플로우 안에서 끝내는 것입니다.

  1. STEP 1정해진 시간에 시작GitHub Actions가 매일 한국 시간 오전 7시에 실행됩니다.
  2. STEP 2AI가 대화문 생성Solar Pro 3가 가족 상황에 맞는 영어와 한국어 뜻을 JSON으로 만듭니다.
  3. STEP 3HTML 파일 저장Python이 날짜별 페이지를 만들고 메인 목록에 링크를 추가합니다.
  4. STEP 4Pages에 공개같은 워크플로우가 결과물을 GitHub Pages에 곧바로 배포합니다.
핵심
생성과 배포를 같은 워크플로우에 둡니다

GITHUB_TOKEN으로 푸시한 커밋은 다른 워크플로우나 브랜치 기반 Pages 빌드를 다시 시작하지 않을 수 있습니다. 따라서 이 가이드는 생성 직후 upload-pages-artifact와 deploy-pages까지 실행하는 구조를 사용합니다.

02

프로젝트 파일 만들기

새 저장소의 루트에 아래 세 파일을 만듭니다. daily/ 폴더는 첫 실행 때 Python이 자동으로 생성합니다.

/ ├── index.html ├── generate_daily.py ├── daily/ # 첫 실행 때 자동 생성 └── .github/ └── workflows/ └── daily_update.yml

1. 메인 목록 페이지: index.html

주석 <!-- DAILY_CARDS_PLACEHOLDER -->는 Python이 새 카드를 삽입하는 기준점입니다. 지우거나 문구를 바꾸지 않습니다.

index.html
참고
재실행해도 카드가 중복되지 않습니다

각 카드에 data-date를 기록하고, Python이 같은 날짜의 카드가 있는지 먼저 확인합니다. 수동 테스트를 여러 번 해도 목록은 한 번만 추가됩니다.

03

AI 호출과 HTML 생성을 담당하는 Python 봇

generate_daily.py는 API 호출, JSON 해석, 날짜별 페이지 생성, 메인 목록 업데이트를 차례로 처리합니다.

generate_daily.py
안전장치이유
환경 변수 확인API 키가 없을 때 모호한 네트워크 오류 대신 즉시 원인을 알려 줍니다.
JSON 범위 추출모델이 코드 펜스를 붙여도 첫 {부터 마지막 }까지 읽습니다.
HTML 이스케이프AI가 만든 문자열이 HTML 태그로 해석되는 것을 막습니다.
날짜 카드 중복 검사같은 날 다시 실행해도 메인 목록에 같은 카드가 쌓이지 않습니다.
모델
현재 공식 예시는 solar-pro3입니다

Upstage의 현재 Chat API 예제는 base_url="https://api.upstage.ai/v1"과 model="solar-pro3"를 사용합니다. 계정에서 제공되는 모델 이름이 다르면 해당 한 줄만 바꾸면 됩니다.

04

매일 실행하고 곧바로 배포하는 Actions 워크플로우

GitHub는 현재 IANA 시간대 지정을 지원합니다. 따라서 UTC를 계산하는 대신 timezone: "Asia/Seoul"과 오전 7시를 그대로 적을 수 있습니다.

.github/workflows/daily_update.yml
UTC
기존 방식으로 연습하고 싶다면

timezone 줄을 지우고 cron: '0 22 * * *'로 바꾸면 됩니다. UTC 22시는 한국 시간으로 다음 날 오전 7시입니다.

워크플로우가 하는 일

  1. 저장소를 체크아웃합니다.

    현재 main의 파일을 Actions 실행 환경으로 가져옵니다.

  2. Python 표준 라이브러리로 API를 호출합니다.

    별도 패키지를 설치하지 않아 의존성 공급망을 줄이고 실행 환경을 단순하게 유지합니다.

  3. 새 대화문을 만들고 커밋합니다.

    변경이 있을 때만 봇 계정으로 커밋하고 main에 푸시합니다.

  4. 공개 파일만 _site에 모읍니다.

    index.html과 daily/만 Pages 아티팩트에 담아 워크플로우나 스크립트가 공개되지 않게 합니다.

  5. 격리된 배포 작업에서 GitHub Pages에 공개합니다.

    콘텐츠 생성 작업은 contents: write만, 배포 작업은 pages: write와 id-token: write만 사용합니다.

안전
Actions는 태그 대신 검증된 전체 커밋 SHA로 고정했습니다

외부 Action이 같은 태그 아래에서 바뀌는 공급망 위험을 줄이고, 생성 권한과 배포 권한도 서로 다른 작업으로 분리했습니다. 버전을 올릴 때는 공식 저장소의 새 릴리스 SHA를 확인해 교체합니다.

05

API 키를 GitHub Secrets에 보관하기

API 키는 HTML, Python, YAML 어디에도 직접 적지 않습니다. 저장소의 암호화된 Secret으로만 전달합니다.

  1. 저장소의 Settings로 이동합니다.

    상단 탭에서 Settings를 엽니다.

  2. Actions Secrets 메뉴를 엽니다.

    Secrets and variables → Actions를 선택합니다.

  3. 새 Repository secret을 만듭니다.

    이름은 정확히 UPSTAGE_API_KEY, 값은 Upstage에서 발급받은 실제 키를 입력합니다.

보안
키가 한 번이라도 커밋되었다면 단순 삭제로 끝나지 않습니다

Git 기록에 남은 키는 노출된 것으로 간주하고 Upstage 콘솔에서 즉시 폐기한 뒤 새 키를 발급받아야 합니다.

06

GitHub Pages를 Actions 방식으로 활성화하기

  1. Settings → Pages로 이동합니다.

    저장소 관리자 권한이 있어야 설정할 수 있습니다.

  2. Build and deployment의 Source를 GitHub Actions로 선택합니다.

    이 가이드의 워크플로우가 직접 Pages 아티팩트를 배포하므로 Deploy from a branch가 아닙니다.

  3. 첫 워크플로우 실행 뒤 주소를 확인합니다.

    프로젝트 저장소라면 보통 https://사용자명.github.io/저장소명/ 형태입니다.

저장소
my-html 저장소의 운영 방식도 같은 원리입니다

참고 저장소는 scripts/generate_index.py로 카테고리 색인을 다시 만들고, actions/upload-pages-artifact와 actions/deploy-pages로 정적 파일 전체를 배포합니다. 이 문서 역시 vibe/에 놓이면 자동으로 Vibe 목록에 포함됩니다.

07

수동 실행으로 전체 파이프라인 검증하기

예약 시간을 기다리지 말고 workflow_dispatch로 한 번 직접 실행합니다.

  1. Actions 탭에서 Daily English Generator를 선택합니다.

    워크플로우 파일이 main에 있어야 Run workflow 버튼이 보입니다.

  2. Run workflow를 누릅니다.

    초록색 완료 표시가 나올 때까지 실행 로그를 확인합니다.

  3. Code 탭에서 생성 파일을 확인합니다.

    daily/오늘날짜.html과 수정된 index.html이 새 커밋에 포함되어야 합니다.

  4. Pages 주소에서 실제 페이지를 엽니다.

    메인 카드, 날짜별 페이지, 목록으로 돌아가기 링크를 차례로 눌러 봅니다.
